# Project_R 开发流程 V2.0

版本：V2.0  
创建时间：2026-06-10  
状态：规划确认稿  
规划目标：从“真实业务 Skill 验证”切回 Project_R 产品基座精修，优先把 Chat、知识库、检索和基础 Agent 做稳定、做清楚、做可验收。

## 1. V2.0 定位

V2.0 不是推翻旧开发流程，也不是继续堆业务工作流。旧 `Project_R 开发流程.md` 保留 Phase 1-10、GBrain 迁移、项目资料质量、9.E 会议 Skill 第一版闭环等历史事实；本文件作为下一阶段主计划。

V2.0 的核心目标：

- 完成 Project_R 作为公司内部 AI 工作台的基本使用体验。
- 把普通 Chat 打磨成稳定可长期使用的日常入口。
- 把知识库查询做成清晰、可信、带引用、可回归的独立能力。
- 把资料检索做成用户可以主动找文件、找来源、找证据的基础能力。
- 把 Agent 做成可控的小执行器：能理解任务、补参、调用明确工具、展示进度、取消/重试，而不是先追求复杂业务自动化。

## 2. 已确认产品边界

### 2.1 保持显式 Chat / GBrain 边界

V2.0 继续采用已确认的路线 A：

- 普通 Chat 不自动查询 GBrain。
- 可信知识问答必须通过显式 `/query`、知识库入口或知识库 Skill。
- Chat 可以提示用户“这个问题建议使用知识库查询”，但不得伪造检索结果或 citation。
- Agent 只有在用户显式启动、选择或授权后，才可调用 GBrain、Skill、文件生成或维护动作。

这与 `CONTEXT.md` 中的 Chat / GBrain Boundary 和 `docs/adr/0016-explicit-chat-gbrain-routing.md` 一致，不需要新 ADR。

### 2.2 V2.0 暂不继续扩张业务工作流

9.E 会议 Skill 第一版已经闭环。后续业务工作流继续保留为增强项，但 V2.0 主线不再优先做：

- 更多行业业务 Skill。
- 复杂会议深水区，例如片段级 ASR 重跑、DOCX 正式会议模板、跨会议说话人记忆。
- 合同/报价/交底文档等新业务闭环。
- 客户情报自动运营或复杂 Agent 自动化。

这些能力不是取消，而是等产品基座稳定后再纳入后续版本。

## 3. V2.0 主线顺序

执行顺序固定为：

1. Chat 基座精修。
2. 知识库查询闭环。
3. 检索与资料发现。
4. 基础 Agent 执行。

理由：

- Chat 是用户日常入口，先稳定交互和会话。
- 知识库是可信回答入口，必须和普通 Chat 有清楚边界。
- 检索是知识库和文件治理的基础体验，支撑后续引用和证据追溯。
- Agent 会放大前面所有不稳定点，因此最后做“可控的小 Agent”。

## 4. 主线 A：Chat 基座精修

目标：普通 Chat 先成为可靠、清楚、可长期使用的桌面 AI 入口。

### A1. 会话与标签页稳定性

- [ ] 标签页、会话、工作区切换不串状态。
- [ ] 切换工作区后，输入框附件、引用文件、Skill 选择和上下文面板不跨工作区残留。
- [ ] 关闭标签页、打开历史会话、新建会话、归档会话的状态一致。
- [ ] Chat / Agent 姿态由全局姿态控制，不被会话标签误绑定。
- [ ] 双对话模式下两侧输入、滚动、附件、模型选择互不污染。

验收：

- Playwright 覆盖：个人工作台、项目工作区、CRM 工作区之间切换，附件引用不串。
- 手工验收：连续打开 5 个会话、切换 3 个工作区、关闭/恢复文件面板，无明显 UI 状态残留。

### A2. 输入框与附件引用

- [ ] 输入框支持稳定的文字输入、换行、发送、禁用态、错误态。
- [ ] 附件引用与项目文件引用分离展示：会话临时附件不伪装成项目资料。
- [ ] 切换工作区、切换会话、新建会话时，未发送附件的保留/清空规则明确。
- [ ] 附件卡片显示来源、大小、可发送状态、不可用原因。
- [ ] 不支持的文件类型给出明确提示，而不是通用“发送失败”。
- [ ] 音视频附件在普通 Chat 中不自动理解；需要用户选择对应 Skill 或 Agent 工作流。

验收：

- 单元测试或组件测试覆盖附件状态机。
- 浏览器点击验证：上传、引用、移除、切换会话后状态正确。

### A3. 消息渲染与生成体验

- [ ] Markdown、表格、代码块、引用来源、复制按钮渲染稳定。
- [ ] 打字机/流式输出不造成对话窗口反复跳动；仅用户贴底时自动跟随。
- [ ] 生成中状态显示真实阶段：等待模型、整理上下文、生成中、保存中、失败。
- [ ] 支持取消生成；取消后可重试，不产生半残状态。
- [ ] 错误消息展示真实可理解原因，区分网络错误、后端错误、模型错误、权限错误。
- [ ] 默认系统提示词不在普通回答上下文卡片中污染展示。

验收：

- Playwright 覆盖长回答滚动、复制代码块、取消生成、失败重试。
- 后端测试覆盖非 ApiError 也能透传错误消息。

### A4. 提示词与模型选择

- [ ] 用户自定义提示词、全局内置提示词、会话级提示词展示层级清楚。
- [ ] 默认 Project_R 系统提示词不作为用户关注信息反复展示。
- [ ] 模型选择在会话中可见、可切换、可持久化，错误时回退策略明确。
- [ ] Skill 与提示词展示分离，不再混成“提示词 / Skill”一行。

验收：

- 选择不同模型发送消息，后端记录和前端显示一致。
- 本轮上下文只展示用户显式选择或有排错价值的 prompt / Skill。

## 5. 主线 B：知识库查询闭环

目标：知识库查询成为独立、可信、可追溯、可回归的能力。

### B1. 显式知识库入口

- [ ] `/query` 路径稳定可用。
- [ ] 提供清晰的知识库查询入口，不要求用户记命令。
- [ ] 普通 Chat 中识别到“应查知识库”的问题时，只建议使用知识库入口，不自动查询。
- [ ] 个人工作台 `/query` 只查 `company-wiki`。
- [ ] 项目工作区 `/query` 查 `company-wiki + 当前项目 source`。
- [ ] 客户工作区 `/query` 只查客户情报 source，不叠加公司或项目 source。

验收：

- 后端测试覆盖三类工作区 source scope。
- UI 验证查询入口和 `/query` 返回一致。

### B2. 引用、gap 和 warning 展示

- [ ] GBrain 返回的 citation 可点击、可预览、可复制。
- [ ] 来源预览显示 source、路径、片段、更新时间和权限提示。
- [ ] gap、conflict、warning 在回答中有清楚展示，不藏在调试信息里。
- [ ] 用户能区分“模型回答”“知识库证据”“系统警告”。
- [ ] 低置信或缺资料时，回答必须诚实提示缺口。

验收：

- 构造有 citation、有 gap、有 warning 的三类 fixture。
- 前端截图或 Playwright 断言展示区域存在。

### B3. 知识库质量回归

- [ ] 公司知识 query 回归保留并纳入常规验证。
- [ ] 项目知识 query 回归覆盖真实项目 source。
- [ ] 客户情报 query 回归覆盖 CRM source scope 和防串库。
- [ ] 回归报告进入管理员面板，显示最近结果、失败用例、趋势和导出。
- [ ] GBrain 不可用时明确显示 service unavailable，不误报空答案为通过。

验收：

- `backend/scripts/gbrain_query_regression.py` 可稳定跑。
- 管理员面板能看到最近质量报告。

### B4. GBrain 服务可用性

- [ ] 后端健康检查能区分 GBrain 未启动、PGLite 初始化失败、source 未注册、embedding 不可用。
- [ ] 管理员能启动/重启 GBrain，并看到 runbook 级错误说明。
- [ ] 普通用户查询失败时看到业务语言错误，不暴露底层堆栈。
- [ ] GBrain runtime home 固定为 `backend/workspace_data/_gbrain/`，不得回写到 source 原始目录。

验收：

- 管理员后台状态页显示 service、source、embedding、sync 状态。
- 人工断开 GBrain 后，前端错误可理解且可恢复。

## 6. 主线 C：检索与资料发现

目标：用户不一定要问 AI，也能主动找到文件、知识、来源和证据。

### C1. 文件检索

- [ ] 项目文件面板支持当前工作区文件搜索。
- [ ] 支持按文件名、路径、类型、状态、上传者、更新时间过滤。
- [ ] 搜索结果可打开所在目录、预览、引用到对话、下载。
- [ ] 回收站、隐藏文件、无权限文件遵守权限边界。

验收：

- 后端测试覆盖搜索权限和过滤条件。
- Playwright 覆盖搜索、预览、引用、返回目录。

### C2. 知识库检索

- [ ] 提供“知识库检索”入口，区别于问答式 `/query`。
- [ ] 检索结果显示标题、source、片段、匹配原因、更新时间。
- [ ] 支持 source scope 过滤：公司、当前项目、客户。
- [ ] 点击结果可打开来源预览或发起 `/query`。
- [ ] 不把普通 Chat 自动升级为知识库检索。

验收：

- 对 company/project/customer 三类 source 做搜索 fixture。
- 结果排序、过滤、空结果提示可验证。

### C3. 引用定位与预览

- [ ] GBrain citation、文件搜索结果、聊天引用文件使用统一来源预览面板。
- [ ] 支持 Markdown、TXT、DOCX 文本预览；PDF/图片/音视频显示可理解占位和元信息。
- [ ] 引用片段能高亮或定位到相关段落。
- [ ] 预览面板关闭后恢复用户原本面板宽度。

验收：

- Playwright 覆盖来源预览打开/关闭、宽度恢复、片段显示。

### C4. 检索质量与排序

- [ ] 项目资料检索按文件类型意图分流，不让低质量会议转写污染非会议问题。
- [ ] Excel、PDF、图纸、邮件、会议各有基础检索权重策略。
- [ ] 命中解释能说明为什么这个结果排在前面。
- [ ] 检索失败能区分没有资料、没有权限、GBrain 不可用、预处理未完成。

验收：

- 复用 8.D 质量回归作为检索主线闸门。
- 每次改排序必须跑相关回归。

## 7. 主线 D：基础 Agent 执行

目标：先做可控的小 Agent，保证用户知道它在做什么、缺什么、会写哪里、如何停止。

### D1. Agent 姿态与任务入口

- [ ] Agent 姿态只在用户显式切换或任务明确需要执行时启动。
- [ ] Chat 姿态下不显示“建议切换 Agent”的错误提示。
- [ ] Agent 任务入口支持：用户文字、当前会话附件、显式引用文件、选定 Skill。
- [ ] Agent 不隐式跨工作区读取文件。
- [ ] 任务开始前明确显示目标、输入、输出位置和权限要求。

验收：

- Agent / Chat 切换不会改变会话标签类型。
- 工作区切换后 Agent 输入不串附件。

### D2. 补参、计划和确认

- [ ] 缺字段时必须显示具体字段名和业务补充指令。
- [ ] 计划卡片展示任务理解、执行计划、待确认项、预计输出。
- [ ] 高成本或写文件动作必须二次确认。
- [ ] 用户可以修改计划或取消执行。
- [ ] 补参状态不应只显示“还需要 1 个字段”。

验收：

- fixture 覆盖缺文件、缺目标目录、缺字段、权限不足。
- 前端断言补参正文包含具体指令。

### D3. 工具 / Skill 调用

- [ ] Tool 层和 Skill 层继续分离：Tool 是通用能力，Skill 是业务包装。
- [ ] Agent 调用 Skill 前显示选中 Skill 和输入来源。
- [ ] Skill 运行中展示真实步骤，不只轮换预设文案。
- [ ] 工具失败返回可理解错误和重试建议。
- [ ] 输出默认先展示，用户确认后才保存到项目/客户文件面板，除非该工作流已有明确例外。

验收：

- audio-transcription、meeting-minutes、meeting-actions 作为样例 Skill 回归。
- 后端测试覆盖无附件、错误附件、权限不足、工具异常。

### D4. 执行状态、取消和重试

- [ ] 长任务有 run id、状态、步骤、日志摘要和结果。
- [ ] 前端可取消仍在运行的任务；后端状态变为 cancelled。
- [ ] 可重试 retryable 失败，不覆盖旧结果。
- [ ] 已完成结果可复制、下载、引用或保存。
- [ ] 失败原因写入审计和通知，不只弹通用 toast。

验收：

- 后端测试覆盖 running/cancelled/failed/completed 状态转换。
- 前端点击验证取消和重试。

## 8. V2.0 不做项

V2.0 明确不做：

- 普通 Chat 自动查 GBrain。
- 新增大批业务 Skill。
- 自动写入公司知识库或自动触发 GBrain 高成本写操作。
- 个人工作台文件库、个人 GBrain source、个人长期记忆。
- 用 Project_R 自建替代 GBrain 的知识图谱、timeline、citation 或 embedding 系统。
- 完整任务系统、日历提醒、跨项目待办管理。
- 自动实体合并、客户画像自动改写、citation-fixer 自动后台写回。

## 9. V2.0 验收总标准

### 9.1 用户体验验收

- 普通员工可以稳定聊天、上传本轮附件、引用项目文件、复制和下载输出。
- 用户能清楚知道什么时候是在普通 Chat，什么时候是在查知识库，什么时候是在执行 Agent。
- 知识库回答有来源、有预览、有缺口提示。
- 检索能帮助用户主动找到资料，不必全靠问答。
- Agent 执行过程可理解、可取消、可重试。

### 9.2 技术验收

- Chat / query / search / agent 四类主路径都有后端测试和前端关键路径测试。
- GBrain 不可用、LLM 不可用、权限不足、文件不存在、source 未同步都有明确错误。
- 不污染真实 `backend/app.db` 和真实 workspace data 的自动化测试。
- `bun run typecheck`、相关 `pytest`、Playwright 关键路径测试通过。
- 任何涉及 GBrain source scope 的改动必须跑 query/source scope 回归。

### 9.3 文档验收

- `Project_R 开发流程V2.0.md` 是 V2.0 主计划。
- `Project_R 开发流程.md` 保留历史事实和旧主线状态。
- 若修改 Chat / GBrain 边界，必须同步 `CONTEXT.md` 和 ADR；当前 V2.0 不修改该边界。
- 若修改前端视觉语言，必须同步 `docs/ui-design-language.md`。

## 10. 推荐执行节奏

### V2.0-1 Chat 基座精修

优先级最高。先修用户每天都会遇到的问题：会话状态、输入框、附件引用、消息渲染、流式滚动、错误、取消、重试。

完成标志：

- Chat 主路径可以连续使用 1 小时不出现状态串联、附件残留、滚动抖动或错误吞没。
- 个人/项目/客户三个工作区的 Chat 边界清楚。

### V2.0-2 知识库查询闭环

完成 `/query` 和知识库入口的产品化。重点是 source scope、引用、预览、gap/warning 和质量回归。

完成标志：

- 用户能用入口完成公司、项目、客户三类知识查询。
- 每个可信回答都有可理解的来源和缺口说明。

### V2.0-3 检索与资料发现

补齐文件搜索、知识库搜索、来源预览和命中解释。

完成标志：

- 用户不用提问，也能找到项目文件、知识库页面和证据片段。
- 搜索结果能安全遵守权限和 source scope。

### V2.0-4 基础 Agent 执行

在 Chat、知识库和检索稳定后，收口 Agent 执行框架。

完成标志：

- Agent 能处理小型可控任务，展示计划、补参、执行、结果、取消、重试。
- Agent 不隐式跨工作区读写，不把普通 Chat 升级为后台动作。

## 11. 当前第一批可执行任务建议

1. 清点 Chat 状态串联问题：工作区切换、会话切换、附件引用、Skill 选择、右侧面板。
2. 为 Chat 输入区和附件引用建立组件/状态机测试。
3. 为 `/query` 建立三工作区 source scope 前端验收脚本。
4. 抽象统一来源预览面板：文件预览、citation 预览、搜索结果预览共用。
5. 为 Agent 补参卡片建立固定 schema：字段名、业务说明、示例、是否必填、来源。
