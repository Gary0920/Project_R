# Project_R 领域术语表

本文档给 Agent 提供 Project_R 的核心领域语言。产品范围、阶段进度和业务 Skill 清单以根目录文档为准：

- `Project_R PRD.md`
- `Project_R 开发流程.md`
- `Project_R 业务工作流清单.md`
- `AGENTS.md`

## 工作姿态 (Work Stance)

Chat 与 Agent 不是会话分类，而是同一会话上的两种**工作姿态**。

- **Chat 姿态**：回答、澄清、起草、解释；只有在用户显式选择 `/query`、知识库入口或知识库 Skill 时，才引用知识库。Chat 可以展示轻量过程状态，但不进入计划审批流。
- **Agent 姿态**：执行、调用 Skill、生成文件、检查项目资料、追问字段、展示执行状态。Agent 可以进入正式计划模式，展示任务理解、执行计划、确认执行、修改计划和步骤进度。

姿态由全局 `activeModeAtom` 统一管理，不是会话属性，不是标签页属性。
标签页是**中立**的，不记录姿态类型。

## 工作台体验整理 (Workbench UI Experience Refresh)

工作台体验整理是 Project_R 前端工作台的视觉、入口、布局、展示层级和交互流程调整。

- 它允许调整通知入口、功能按钮位置、Agent 计划展示、执行进度、引用来源呈现和客户情报信息架构。
- 它不改变后端功能、权限规则、数据流、GBrain source scope 或业务能力边界。
- 它不是单纯换皮，也不是功能重构。
- 视觉方向可以提取原型 A 版的柔和手感，包括圆角、字号、行高、间距、高度、低对比细边框和浅色柔和感，但不继承 A 版整页布局。
- 卡片只用于消息、重复项、弹窗和真正需要框住的工具，不把页面区块都做成卡片。
- 右侧区域采用单一活动上下文面板，不同时堆叠多个主要面板。
- 右侧上下文可在来源详情、文件管理、CRM 文件管理、客户画像辅助资料、GBrain 状态/管理员处理和双对话之间切换。
- 顶部或上下文工具按钮负责切换右侧面板；点击引用、文件或 CRM 对象可以主动打开对应右侧详情。
- 右侧面板状态应绑定当前标签或当前工作区，避免切换会话后继续显示旧来源。
- 业务功能入口使用图标加短文字，例如 CRM、文件管理、客户情报、来源详情、GBrain 状态和双对话。
- 系统工具入口可以只使用图标，例如通知、设置、用户账号、更新和窗口控制，但必须提供 tooltip、状态徽标或未读数。
- 新用户不熟悉的业务功能不得只靠抽象图标表达。
- 工作台体验整理应分阶段实施：第一阶段先做壳层和核心体验，第二阶段再做 CRM 深水区。
- 第一阶段包括视觉 token、顶部导航、右上系统工具区、左侧工作区收纳选择器、右侧单一活动上下文面板框架、通知中心、轻量引用来源和 Agent / Chat 过程展示。
- 第二阶段包括 CRM 总览、具体客户画像详情、关系图谱、时间线、CRM 文件管理、GBrain 状态/管理员处理和客户情报真实数据适配。
- 第一阶段必须同步更新 `docs/ui-design-language.md`，把旧 Proma 强约束改为新的 Project_R Workbench 规则后再落实现有前端。
- 第一阶段不深入改造 CRM 真实业务页面，只保留顶部 CRM 入口、面板框架、占位或跳转边界；CRM 总览和具体客户画像真实数据适配进入第二阶段。

## Chat / GBrain 边界 (Chat / GBrain Boundary)

Project_R 的普通 Chat 与 GBrain 知识库是互相独立的能力，不做隐式自动联动。2026-06-04 决策：普通 Chat 的业务纪律、语气和回答边界由 Project_R 内置提示词、全局底层提示词、用户自定义提示词和会话级提示词约束；可信知识库问答由显式 `/query`、知识库查询入口或明确选择的知识库 Skill 路由到 GBrain。

- 普通 Chat 不自动查询 GBrain，不自动读取 `company-wiki`、项目 source 或客户 source。
- 普通 Chat 可以基于当前用户输入、当前会话附件、显式引用文件和提示词进行回答、改写、解释、起草和建议。
- 当用户问题需要公司事实、项目事实或客户事实，但没有显式选择 `/query` 或知识库入口时，Chat 应提示用户切换到知识库查询，而不是伪造检索结果。
- `/query` 是正式知识库问答入口；它必须通过 Project_R 后端权限判断后，按当前工作区和 source scope 调用 GBrain，并返回可追踪 citation / gap / warning。
- 个人工作台中的 `/query` 只查询公司全局知识库 `company-wiki`，不查询任何项目 source 或客户 source。
- 项目工作区中的 `/query` 查询 `company-wiki + 当前项目 source`：公司全局规则、通用流程和跨项目沉淀来自 `company-wiki`，当前项目特有资料、会议、邮件、图纸和项目事件来自当前项目 source。
- 客户工作区中的 `/query` 只查询客户情报 GBrain 数据，不叠加 `company-wiki` 或项目 source。客户情报 GBrain 数据来自 `workspace_data/customer/` 下客户往来邮件、会议记录、联系人、公司、项目关系等资料经 Project_R 提炼并由 GBrain Entity Enrichment 吸收整理后的结果；`customer-reference` 如仍出现在代码或文档中，只视为早期实现 source id，不是产品层术语。
- Agent 姿态或业务 Skill 只有在用户显式启动、选择或授权时，才可以调用 GBrain 查询、同步、提炼或维护能力；不得把普通闲聊升级为后台知识库动作。
- 若后续要增加自动意图识别、自动知识库推荐或 Chat 自动检索，必须作为新的产品决策单独确认，并更新本文档和 ADR。

## 标签页 (Tab)

标签页是会话或功能面板的打开句柄，属性只包含：
- `id`、`sessionId`、`workspaceId`、`title`

不包含 `type`（chat/agent），不绑定姿态。
历史对话单击默认复用当前标签；只有中键、Ctrl 点击或右键菜单“在新标签页打开”才创建新标签。

## 工作区主页 (Workspace Home)

工作区主页是当前工作区没有打开标签或没有对话时显示的真实工作起点。

- 允许关闭所有标签页。
- 工作区主页包含输入框、最近对话、项目文件入口和项目目录入口。
- 在主页发送首条消息默认创建 Chat 对话；若首条消息触发 Agent、Skill、文件生成或多步骤执行意图，则自动切换到 Agent 姿态。

## 项目目录面板 (Project Directory)

项目目录面板是搜索、打开和新建公司项目工作区的统一入口。

- 入口显示为“项目目录”。
- 公司项目按 `AURA`、`BFI`、`SPECWISE`、`SYNOVA` 品牌分组。
- 新建公司项目时必须选择品牌，再填写项目代号。

## 工作区 / 项目 (Workspace / Project)

工作区是 Project_R 的软件使用容器；在业务上，每个工作区一一对应一个项目，通常以项目代号命名。

- 用户界面可使用“工作区”表达软件操作入口，业务语境使用“项目”表达资料归属。
- 代码内部历史命名继续使用 `workspace`，不代表业务上存在独立于项目的另一类容器。
- 后端创建工作区时同步创建系统管理的项目资料目录。
- Project_R 后端统一资料源根目录为 `workspace_data/`，不额外新增同级 `knowledge_sources/` 目录。
- 公司全局知识库、公司项目资料和客户资料属于 Project_R 后端管理的业务资料源，按子目录和 source scope 隔离；用户个人资料只作为本机选择文件按次授权，或作为会话临时附件进入当前对话上下文，不进入个人文件产品线。
- 公司全局知识库源文件位于 `workspace_data/global/company-wiki/raw/`，只存放管理员投放的原始源文件。
- 用户源文件目录不再存放 GBrain 派生 Markdown 或过程文件；旧 `workspace_data/global/company-wiki/derived/` 只代表早期 MVP 实现和历史数据，不再作为目标架构扩展方向。
- Project_R 预处理产物统一位于 `workspace_data/_preprocessed/`；公司全局知识的目标 GBrain source repo 是 `_preprocessed/company/company-wiki/gbrain-ready/`，过程文件在 `_preprocessed/company/company-wiki/runs/`，状态和来源映射在 `_preprocessed/company/company-wiki/manifests/`。
- GBrain-ready Markdown 默认不是普通人工编辑区；人工修正应优先通过源文件修订、知识审核、citation-fixer 或专门纠错流程形成可追溯变更。
- GBrain-ready source repo 或等价 manifest/audit-log 机制用于审计、对比和回滚；不默认配置远程仓库，不上传 GitHub，不进入 Project_R 主代码仓库。
- 公司项目文件只能存放在系统管理的项目资料目录内：公司项目位于 `workspace_data/project/{品牌}/{项目代号}`。
- 品牌级目录固定为 `AURA`、`BFI`、`SPECWISE`、`SYNOVA`，品牌下一级才是具体业务项目。
- 公司项目资料目录默认包含 `01-合同与报价`、`02-图纸与技术资料`、`03-会议纪要`、`04-变更与签证`、`05-生产与发货`、`06-现场与客诉`、`99-未归档文件`。
- 项目工作区右侧文件面板默认显示项目资料根目录，不按会话自动定位。
- 公司项目默认是开放项目：所有用户都可搜索、进入、上传项目文件，并在项目对话中使用该项目资料。
- 项目成员列表不代表开放项目的全部访问者；成员列表用于记录项目管理员、隐藏项目人员白名单和需要显式通知/治理的协作人。
- 项目成员可上传文件；普通访问者和成员只能删除自己上传的文件；项目管理员可删除所有文件。
- 项目文件移动在用户界面中采用 Windows 风格的剪切 / 复制 / 粘贴或单项拖拽，不再要求用户输入内部目标路径；第一版不做多选。
- 删除进入该工作区自带的 `.trash` 回收站；`.trash` 在普通文件目录中显示为回收站入口，用户点击后进入回收站视图查看和还原。
- 项目资料只影响对应项目内的项目对话，不影响其他项目或无项目会话。
- 可进入该项目的用户，在项目对话中可使用该项目资料。
- 项目资料不能覆盖公司全局底层规则。
- 项目资料“一键录入项目知识库”只允许系统管理员或该项目的项目管理员使用；普通项目访问者和普通成员不能触发一键录入。

## 业务文件容器 (Business File Container)

2026-06-04 决策：Project_R 中需要长期保存、共享、权限治理、审计、入库、复盘或被项目/客户知识库引用的业务文件，只能进入项目工作区或客户工作区的文件面板；个人工作台不再作为业务文件容器。

- 项目资料和客户资料是正式业务文件容器，承接上传、预览、引用、移动、删除、回收站、权限、审计、GBrain source 状态和一键入库。
- 个人工作台只承接个人对话入口、提示词上下文、会话临时附件和本地导出/下载动作，不承接长期业务文件管理。
- 本机选择文件、剪贴板粘贴文件和会话附件默认只服务本轮或当前会话；它们不是项目资料、客户资料或公司知识。
- 个人工作台中的临时文件和生成结果默认只作为本轮结果展示，可复制内容或下载到本地；不提供“保存到项目资料”或“保存到客户资料”的跨工作区沉淀入口。
- 项目相关工作应在项目工作区完成，客户相关工作应在客户工作区完成；只有在对应工作区内生成或上传的业务资料，才进入该工作区的权限、审计、回收站和 GBrain source 规则治理。
- 2026-06-04 决策：项目/客户工作区里的业务 Skill / Agent 输出也采用“先展示、用户确认后保存”的逻辑。生成结果默认先作为本轮结果展示；用户确认保存后，才写入当前工作区文件面板，默认位置为当前工作区 `99-未归档文件`。
- 项目/客户工作区内的 Skill 输出保存后，只进入该工作区的文件权限、审计、回收站和后续入库候选范围；保存不等于自动入库 GBrain，GBrain 入库仍需要显式入库动作或另行确认的自动规则。
- 项目/客户工作区内，用户确认保存后的 Skill / Agent 输出是普通工作区文件；如果文件类型被当前预处理 Skill 支持，就可成为该工作区的待录入候选。
- 导出和下载默认面向本机文件系统；Project_R 不提供“导出到个人工作台”“下载到个人工作台”或用个人工作台代替本机保存位置的链路。
- 任何后续 UI、后端接口或测试不得重新引入个人文件面板、个人业务文件库或个人 GBrain source，除非先形成新的产品决策和 ADR。

## 客户情报资料根 (Customer Intelligence Root)

`workspace_data/customer/` 是客户情报和 CRM 场景资料根，不是 `workspace_data/project/` 的另一种项目目录。它主要保存客户往来邮件、会议记录、联系人资料、公司资料、客户项目关系、沟通事件和销售判断线索，用于构建客户画像。

- 目标用户主要是销售团队和需要做客户关系管理、客户风险判断、关键人判断、公司关系判断、项目关系判断的业务人员。
- “CRM 客户情报面板 / 客户画像面板”是业务情报视图，不是文件容器，也不是 GBrain 管理控制台。它面向销售和客户管理用户，以业务语言展示客户实体、关系、近期互动、事件详情和来源追溯。
- 客户画像入口应作为主工作台独立入口与“工作区资料”并列；工作区资料只承担源文件治理，客户画像承担 GBrain-derived 客户情报的阅读和判断。
- CRM 客户情报入口默认进入 CRM 总览，展示全公司客户列表、重点客户、近期互动、风险/机会、待处理资料和客户情报 source 状态摘要。
- 具体客户画像只在用户选中或搜索某个客户后出现，展示该客户的摘要、关键联系人、关系图谱、时间线、来源证据和 GBrain 状态。
- 右侧客户资料面板可以按当前选中的客户或视图过滤资料，但资料仍来自同一个 `workspace_data/customer/CRM/raw/` 客户情报资料根。
- UI 主入口应避免使用“客户资料”指代文件治理，以免和客户画像/客户情报混淆。
- 上传、文件夹、录入、回收站、文件状态和文件权限治理统一使用“CRM 文件管理”或“文件管理”表达。
- CRM 应作为独立功能面板显示，不继续增加到对话侧边栏；对话侧边栏主要承载工作区、会话、Chat / Agent 姿态和历史对话。
- CRM 独立功能面板入口位于顶部业务导航层，靠近当前工作区导航；它不是右上角设置/通知/账号这类全局系统工具。
- `workspace_data/customer/` 的目标不是按客户复制项目工作区文件治理模型，而是向 GBrain 投喂客户情报资料，让 GBrain Entity Enrichment 自动提炼实体、关系和事件。
- `workspace_data/customer/CRM/raw/` 是当前全公司 CRM 客户情报源文件入口；CRM 不是项目品牌层级，也不是某个单客户 workspace 的子目录。用户在 Project_R 中点击 CRM 即进入服务全公司营销需求的客户情报工作区。
- CRM 的 GBrain-ready 预处理目标固定为 `workspace_data/_preprocessed/customer/crm/gbrain-ready/`，过程文件为 `workspace_data/_preprocessed/customer/crm/runs/`，manifest 为 `workspace_data/_preprocessed/customer/crm/manifests/`；不得使用 `_preprocessed/customer/{workspace_id}-crm/` 这类项目化层级。
- 目标图谱包括 People Graph、Company Graph 和 Project Graph，用于围绕某个客户、某家公司、某个项目中的人或组织关系做决策评估。
- 客户资料进入 GBrain 后，正式查询对象是 GBrain 已经精炼、吸收、整理后的客户情报数据，而不是原始邮件/会议文件本身，也不是公司全局 `company-wiki`。
- 客户工作区 `/query` 只查询客户情报 GBrain 数据；不叠加 `company-wiki`，不查询项目 source，不按“公司全局 + 客户”组合。
- 客户情报是受限业务情报，不是公司公共知识库；权限由 Project_R 客户工作区成员、组别和系统管理员规则控制。
- `customer-reference` 如仍用于当前代码、manifest 或测试，只能视为早期 MVP 验证 source id，不是正式质量基线；后续应清理旧 GBrain source/client 和旧预处理产物，保留原始 Markdown 资料源，用 GBrain 原生客户画像 / Entity Enrichment 能力重跑。

## 开放项目 (Open Project)

开放项目是公司项目工作区的默认可见性。

- 所有有效用户都能在项目目录中搜索到开放项目。
- 所有有效用户都能进入开放项目并使用项目资料。
- 开放项目仍可设置项目管理员；项目管理员负责项目名称、隐藏状态、人员白名单、组别白名单和项目文件治理。

## 隐藏项目 (Hidden Project)

隐藏项目是敏感、不公开或仅部分团队可见的公司项目工作区。

- 隐藏项目不会被未授权普通用户搜索到。
- 隐藏项目只允许系统管理员、项目成员白名单用户、授权组别内用户进入。
- 隐藏项目的人员和组别白名单由系统管理员或项目管理员在成员管理面板维护。

## 组别 (Work Group)

组别是用户在公司现实组织中的团队或职责归属，例如销售部、设计部、管理层。

- 组别由系统管理员在用户管理中维护。
- 组别用于隐藏项目的访问白名单，不等同于系统角色。
- 用户只有一个当前组别；跨组项目访问应通过隐藏项目白名单中的具体人员或后续多组别能力处理。

## 个人工作台 (Personal Workspace)

个人工作台是 Project_R 后端按用户隔离的个人对话工作区，显示名为 `{用户名}的工作台`；用户名由软件根据当前登录用户自动识别。它不是公司项目、不是本地文件夹、不是个人长期知识库，也不是个人记忆系统。2026-06-04 决策：个人工作台冻结为个人上下文入口；用户个性、写作偏好和常用工作方式优先通过用户自定义提示词、会话级提示词和本轮显式引用文件表达，不通过个人 GBrain source 或长期个人记忆实现。

- 每个用户都有一个个人工作台，可见且置顶，不可删除、不可重命名、不可归档。
- 左侧工作区列表应分为 `个人` 和 `项目` 两组；`个人` 组只显示当前登录用户的 `{用户名}的工作台`，`项目` 组显示公司项目工作区。
- 个人工作台使用个人图标，项目工作区使用项目或文件夹图标。
- 个人工作台的中间对话区域和项目工作区一致，支持会话、Chat / Agent 姿态、输入框、附件、引用、导出和消息气泡；差异主要在项目上下文、知识库入口和文件面板能力。
- 2026-06-04 决策：个人工作台不再提供右侧个人文件面板。右侧常驻文件面板只用于项目资料、客户资料等共享/受限资料空间。
- 2026-06-04 决策：个人工作台允许运行轻量业务 Skill / Agent 生成类任务，例如根据本轮输入、会话附件、本机按次选择文件或用户填写字段起草邮件、改写文本、填写模板、生成可下载文件。
- 个人工作台中的轻量业务 Skill / Agent 只能使用本轮显式授权的输入和临时上下文；输出默认只作为本轮结果展示，可复制内容或下载到本地，不提供保存到项目/客户资料的跨工作区动作。
- 个人工作台不承载 GBrain 入库、长期资料整理、项目/客户 source 治理、个人文件库治理、项目复盘、客户画像、图谱实体合并或知识审核。
- 个人工作台不属于公司项目，不显示在公司项目目录面板中，也不应在项目文件面板中标成“项目参考文件”。
- 个人工作台不需要成员管理、隐藏/开放状态或项目管理员；它只归当前登录用户使用。
- 个人工作台不保存“常用文件”或“对话文件”作为产品级长期文件区；用户要长期保存导出、下载或生成文件时，应使用本机文件系统。
- 导出与下载统一使用“导出到本地”或“下载到本地”。不再提供“导出到工作台”或“下载到工作台”。
- 会话附件是用户本轮手动上传、粘贴或拖拽提供的临时上下文；它应显示在输入框预览和消息气泡中，不进入个人文件面板。
- 会话附件可以有后端临时存储和清理机制，但该机制是技术实现，不是个人文件产品线。
- 文件不会被 AI 默认读取；无论文件来自本机选择文件、会话附件还是项目资料，用户都必须显式引用、发送、分析或选择知识库查询后，AI 才能读取对应文件或文件片段。
- 文件右键功能面板应提供“引用文件”选项；引用后，输入框顶部显示引用预览，用户可在发送前查看引用对象并移除引用。
- 引用文件默认只作用于本轮消息；消息发送后，该引用不自动延续到后续消息。
- 如果后续需要持续引用当前会话，应另行提供“固定引用到当前会话”之类的显式操作，但不能作为默认行为。
- 输入框预览区必须区分“附件”和“引用文件”：附件表示本轮要上传或发送的文件，引用文件表示本轮只读取已存在于项目资料、客户资料或其他可访问位置的文件。
- 输入框工具栏只保留一个附件按钮：桌面端点击后选择本机文件并进入发送前授权流程；Web-only 或本地 Worker 不可用时，点击同一按钮降级为会话临时上传。不得把会话临时附件和本机选择文件拆成两个相邻图标。
- 输入框预览区中每个文件都显示为独立卡片；第一版只承诺简单图片和 PDF 预览，Office 文件和其他通用文件只显示文件卡片与基础信息。
- 消息发送后，用户消息气泡中应保留本轮附件和引用文件的紧凑卡片摘要；图片保留缩略图，PDF 和通用文件保留文件卡片。
- AI 消息如果实际读取了附件或引用文件，应在回答正文或来源区显示紧凑来源标记，并可定位到对应用户消息文件卡片或打开文件预览。
- AI 回答的文件来源采用两层显示：正文内可插入紧凑来源标记做就近说明，消息底部保留统一“引用来源”列表集中展示本次实际读取的文件。
- 聊天消息底部的“引用来源”默认是轻量单行列表，不默认展开大卡片、长摘要、metadata 或 chunk 内容。
- 用户点击来源编号或来源名后，右侧来源详情面板展示 citation 片段、文件路径、source、时间以及相关 gap / conflict / warning。
- gap / conflict / warning 在聊天流中只显示轻量提示，详细内容进入右侧来源详情；管理员或调试视图可提供技术详情展开。
- 项目资料文件面板承担共享项目文件管理、引用、预览、一键录入项目知识库、权限、审计和项目 source 状态展示；个人工作台不复用该文件面板。
- 公司知识库建设是整体性的管理员/组织流程，不由个人工作台文件影响；个人工作台文件不能直接或间接提交为公司知识候选，不能进入公司知识库，也不能进入 GBrain `company-wiki`。
- 如个人材料确实需要成为公司知识，应脱离个人工作台路径，由管理员按公司知识录入流程重新收集、审核和入库。
- 个人工作台面向单用户，不设计多人协作、成员权限或共享审计；系统管理员也不把其他用户的个人工作台当作项目空间浏览。
- 个人工作台内的文件默认不进入公司知识库，不进入项目 source，也不进入 GBrain。
- 个人工作台默认不承担长期知识沉淀、跨会话个人记忆或个人 GBrain source 职责；资料只有在用户显式授权后，才可复制进入项目资料，不能进入公司知识治理流程。
- `workspace_data/user/` 不再作为个人文件区创建或使用；个人工作台没有后端文件面板。会话附件如需后端临时存储，应走独立附件临时路径和清理机制，不能重新变成个人长期资料目录。
- 个人工作台普通聊天不自动查询公司知识库；用户显式使用 `/query` 或知识库查询入口时，只查询公司全局 `company-wiki`。
- 个人工作台不是公司项目，默认没有项目 source，也不能查询项目/客户 source；只有项目工作区的 `/query` 才能叠加当前项目 source。
- 用户从本机任意路径手动选择文件时，这些文件只是“本机选择文件”，不需要先放入固定的本地私人空间根目录。
- 本机选择文件只有在用户确认发送、分析或保存时，才上传必要的文件或内容片段到会话临时上下文或项目资料。
- 个人工作台对话与项目对话相互独立，个人工作台没有“当前项目”上下文。
- 个人工作台中的会话附件或本机选择文件不提供“保存到项目资料”动作；如果资料属于某个项目，用户应切换到对应项目工作区后上传或生成，默认进入该项目 `99-未归档文件` 并按项目权限、审计、`.trash` 回收站和 GBrain 项目 source 规则治理。

## 系统管理员 (System Administrator)

系统管理员是拥有 Project_R 全局后台管理权限的用户角色。

- 系统管理员用于处理用户管理、全局知识审核、系统健康告警、LLM Key/成本告警和全局配置类事务。
- 系统管理员不等同于某个项目内的工作区管理员。

## 工作区管理员 (Workspace Administrator)

工作区管理员是某个项目资料容器内拥有管理权限的成员角色。

- 工作区管理员用于处理对应项目内的文件管理、成员协作和项目级风险通知。
- 工作区管理员不自动拥有系统后台管理权限。

## 会话临时附件 (Session Attachment)

会话临时附件只服务当前会话，是用户在聊天中临时提供的参考材料。

- 不自动进入项目资料库。
- 可通过附件按钮、剪贴板粘贴或拖拽到对话区添加。
- 统一附件按钮、剪贴板粘贴和拖拽添加的是本轮临时上下文；桌面端通过该按钮选择本机文件时，必须在发送前显示来源和授权说明。Web-only 或本地 Worker 不可用时，同一按钮降级为会话临时上传。
- 默认按用户和会话隔离，存放在后端临时附件目录或后续本地 Worker 授权缓存中；该存储不能被解释为个人文件面板或长期私人资料库。
- Web-only 或本地 Worker 尚未启用的过渡期，可使用后端临时附件目录承接上传，并按会话附件保留策略清理。
- 删除会话时默认只删除会话记录；附件清理按会话附件保留策略处理，不提供个人工作台回收站语义。
- 文本类附件和可提取文本的 PDF 可进入本次对话上下文。
- 图片附件在支持图像输入的 MiMo 模型下会以多模态 content block 投递给模型；DeepSeek 等不支持多模态的模型会阻止发送并提示切换模型。
- 视频、音频、OCR 和不可提取文本的扫描 PDF 仍属于后续能力，当前不会静默当作可解析内容使用。
- 后续如需沉淀为项目资料，用户应在项目工作区内重新上传、生成或执行项目 Skill，默认进入当前项目的 `99-未归档文件`。

## Agent 执行架构 (Agent Execution Architecture)

Project_R 采用混合 Agent 架构：后端是 Agent 控制面，执行能力按数据归属分流。

- Agent 控制面由 Project_R 后端承担，负责登录身份、项目权限、GBrain source scope、模型配置、AgentRun / AgentJob 状态、审计日志和通知。
- Server Agent Worker 处理公司知识库、项目资料、业务 Skill、项目文件生成、GBrain ingest / query / think 等需要共享、审计或后端模型 Key 的任务。
- Local Agent Worker 处理用户按次选择的本机文件读取、本地预处理、Office/桌面类自动化和用户私有草稿。
- 前端负责用户操作、预览、确认、授权和状态展示，不是公司级权限判断的最终权威。
- 文件不会被 AI 默认读取；无论文件来自本机选择文件、会话附件、项目资料还是客户资料，用户都必须显式引用、发送、分析或选择知识库查询后，AI 才能读取对应文件或文件片段。
- 数据在哪里，Agent 就优先在哪里处理；跨越本机选择文件、会话临时附件、项目资料、客户资料、公司知识边界时必须产生显式授权和可追踪记录。
- 后端长任务不得阻塞普通 HTTP 请求；项目摄取、复杂文件解析、视频转写、批量文件生成和项目复盘应进入 AgentJob / Worker 队列。
- 后端并发能力靠任务队列、Worker 池、限流、任务状态和通知中心扩展，不靠把所有 Agent 执行塞进 FastAPI 请求线程。
- 本地 Worker 不绕过后端权限：当任务读取项目资料、写入项目文件、查询公司知识库或调用公司模型 Key 时，仍必须通过后端控制面授权。

### Local Agent Worker MVP 边界

Local Agent Worker 第一版只负责打通私人资料边界，不做完整自主执行 Agent。

- 第一版能力白名单：读取用户显式选择的本机文件、生成图片缩略图和基础预览、提取 Markdown / TXT / 可读 PDF 文本、生成本地摘要或待发送片段、把用户确认后的片段或文件交给后端任务。
- 第一版必须支持发送前授权：前端显示本地文件名、类型、大小、来源范围、将发送的内容形式、目标范围和保留策略，用户确认后才允许跨越到后端。
- 文本类本机文件默认优先发送片段或摘要；只有用户明确选择“发送原文件”时才上传原始文件。
- 图片类私人附件用于多模态模型时必须提示图片将被发送到后端模型链路；发送后的聊天记录应能查看已发送图片和来源标记。
- 个人工作台不提供保存到项目资料的复制动作；项目资料上传、生成和治理应在项目工作区内完成。
- Local Agent Worker 第一版禁止扫描整块硬盘、后台自动上传、默认同步本机文件、执行任意 shell 命令、绕过后端调用公司模型 Key、直接写项目目录、直接写公司知识库。
- 本地 Worker 不可用时，产品可以降级为后端临时附件上传，但必须用界面文案区分这是“临时上传处理”，不是本机文件长期同步。

## 全局底层规则 (Global Base Prompt)

`backend/prompt_presets/global-base-prompt.md` 是公司级全局底层规则文件。

- 文件可为空，空文件不影响现有行为。
- 有内容时由后端强制注入到 system prompt 最前面。
- 普通用户不可切换、关闭或编辑。
- 优先级高于会话提示词、Agent 模式提示、附件、项目资料、知识库和用户输入。

统一上下文优先级：

1. 全局底层规则
2. 会话选择提示词 / Agent 模式提示
3. 会话临时附件
4. GBrain source 检索结果 / 当前项目资料
5. 用户问题

## 会话提示词 (Session Prompt)

会话提示词是当前会话的 system prompt 选择，不是输入框模板。

- 来源分为 Project_R 内置提示词、公司预设提示词、用户本机自定义提示词。
- Project_R 内置提示词只读；公司预设提示词由后端保存；用户自定义提示词保存在 Electron `userData` JSON，不进 Git、不上传后端。
- 选中提示词后只在输入框上方显示 chip，不改写输入框正文。
- 每个会话独立选择提示词，新会话默认使用 Project_R 内置提示词。

## 文本变换类提示词 (Text Transformation Prompt)

文本变换类提示词用于处理用户已提供的文本本身。

- 典型任务包括改写、润色、翻译、压缩、扩写、格式整理和语气调整。
- 文本变换类提示词不是知识库问答，也不是项目资料分析。
- 选中文本变换类提示词时，系统默认不引入公司知识库、项目资料或外部规范。
- 用户明确使用 `/query` 或明确要求查询公司资料、知识库、项目文件时，才切换为知识库问答。
- `/query` 是 Project_R 软件内“查询知识库”Skill 调用指令；普通 Chat 保持 chatbot 能力，不自动查询 GBrain。
- 文本变换类提示词不得新增用户原文没有的事实、标准号、文件名、分类、建议或结论。

## 模型配置档 (Model Profile)

`model_profile` 是用户可见的模型路由配置档，不是 API Key。

- 后端通过 `LLM_MODEL_PROFILES` 等配置把 profile 映射到 provider、真实模型名、描述文案和私有参数。
- 多个 profile 可以共享同一组 provider Key，例如 DeepSeek Flash / Pro 共享 DeepSeek Key，MiMo V2.5 / V2.5-Pro 共享 MiMo Key。
- 前端模型下拉只显示 `/health/llm` 返回的已配置 profile；未配置 API Key 的模型不显示、不占位。
- 前端不得暴露 API Key、Provider 切换、多 Key 配置或厂商私有参数。

## 思考模式 (Thinking Mode)

输入区“思考”按钮是参数开关，不是模型切换。

- 前端发送 `thinking` 布尔值。
- DeepSeek 由后端生成 `thinking.type` 与 `reasoning_effort`。
- MiMo 由后端生成 `thinking.type`。
- 推理火力大小保留在后端环境变量或 profile 配置中，不作为员工端 Key/Provider 配置暴露。

## 知识库查询入口 (/query)

`/query` 是 Project_R 的系统级知识库查询入口，不是普通企业业务 Skill。

- 普通 Chat 默认保持大模型对话能力，不自动查询 GBrain。
- 用户显式使用 `/query`、选择知识库查询入口，或通过等价的知识库查询 UI 入口时，才进入 GBrain 知识库查询链路。
- `/query` 可在界面上像能力选择一样被展示和取消，但不归入普通企业业务 Skill 列表，不由普通 Skill 治理流程管理。
- `/query` 的作用是把当前问题限定到已授权的公司知识库或当前项目 source，并要求回答带有可追踪来源。

## 知识库 (Markdown Wiki / GBrain)

Project_R 的知识库产品边界是：GBrain 是知识库内核，Project_R 是公司业务入口、权限审计、原始资料保管、GBrain 友好 Markdown 预处理、信息检索转发、知识审核和可视化工作台。

- `backend/knowledge_base/wiki/` 是当前公司全局知识库的既有 Markdown 资产；这些文件来自 Obsidian 复制导入，属于开发阶段临时资产，不作为 GBrain 接入后的权威知识库。
- 公司知识库可在 GBrain 架构接入后，从重新投喂的 Obsidian 源文件或其他源文件重新摄取生成。
- 旧 Project_R RAG 不做迁移：`knowledge_base/wiki`、Wiki Router、Chroma/vector_store 不作为 GBrain 的回退层或权威输入；由管理员重新投喂源文件生成 `company-wiki`。
- GBrain 负责 source sync/import、索引、schema、entity enrichment、graph、timeline、hybrid search、综合回答、citation 和 gap analysis。
- Project_R 不修改 GBrain 成熟架构，不替代 GBrain schema、graph、entity enrichment 或 timeline；优先使用 GBrain 原生 command、operation、skill、recipe、schema pack 和 Admin 能力。
- Project_R 的知识库、检索、摄取、纠错、维护、Skill 和 Agent 相关能力采用 GBrain-native first 原则：先确认 GBrain 是否已有原生 command、operation、skill、recipe 或 schema 机制，再决定 Project_R 是否只做 adapter 和 UI。
- Project_R 不重复建造 GBrain 已经具备的复杂知识库核心能力；只有 GBrain 缺口明确、且无法通过配置、adapter 或 Skillify 补齐时，才在 Project_R 内新增能力。
- GBrain 生成的派生 Markdown 必须保持可审阅、可备份、可迁移；运行时索引不能成为唯一不可见事实源。
- Project_R 可以作为 GBrain 知识库的显示载体，但不能退化成纯前端壳；身份权限、项目边界、审计、原始文件生命周期和知识审核仍属于 Project_R 的业务责任。
- Project_R 是 GBrain 摄取源文件的管理入口；用户和管理员只需要把源文件放入 Project_R 管理的公司、项目或客户资料位置。
- Project_R 后端统一使用 `workspace_data/` 作为资料源根目录；公司全局知识、项目资料和客户情报分别按 `global/`、`project/`、`customer/` 子目录隔离。`project/` 首层目录是动态品牌大类，默认包括 `AURA`、`BFI`、`SPECWISE`、`SYNOVA`，真实测试品牌目录为 `TEST`；品牌下一级才是具体项目工作区。`customer/` 是 CRM 客户画像资料根，不是项目目录；CRM 源文件入口固定为 `workspace_data/customer/CRM/raw/`。`user/` 子目录不再作为个人文件区创建或使用。

## 资料预处理层 (Source Preprocessing Layer)

资料预处理层是 Project_R 在源文件与 GBrain 之间的适配层。

- 目标是把各种格式的源文件预处理成 GBrain 友好的 Markdown，最大化 GBrain 对事实、实体、关系、时间线、引用和不确定性的吸收质量。
- 源文件不被修改；源文件删除也不自动删除 GBrain-ready Markdown 或 GBrain 知识库内容。
- 源文件修改后不自动重跑；Project_R 标记 `source_changed` / `needs_repreprocess`，由用户或管理员显式点击“录入”后创建新 run。
- 用户在工作区文件面板点击“录入”时，默认递归处理当前打开路径及子文件夹内的待处理文件，并必须二次确认递归范围、文件数量、类型和高成本模型提示。
- 文件右键菜单提供“录入此文件”，用于单文件预处理或重跑；第一版不做复杂多选批量。
- 一次“录入”生成一个 `preprocess_run_id`；过程文件按 run 保存在 `runs/{run_id}/`，最终 GBrain-ready Markdown 按源文件稳定路径写入 `gbrain-ready/`。
- 待处理状态包括 `new`、`source_changed`、`failed_retryable`、`pending_capability_now_supported`；`synced`、`processing`、`source_deleted`、`failed_non_retryable`、`ignored` 不默认处理。
- 第一版人工动作只提供“忽略录入 / 取消忽略”和“重新录入此文件”；其他状态由系统根据 hash、run、manifest 和 GBrain sync 结果维护。
- 项目工作区中，系统管理员和工作区管理员可录入文件夹和单文件；普通成员第一版只允许录入自己上传的单个文件。客户工作区中，只有系统管理员和客户工作区管理员可触发录入。公司全局知识库只允许系统管理员录入。
- 预处理成功写入 `gbrain-ready/` 后，Project_R 自动触发当前 source 的 GBrain sync；但不自动触发 Entity Enrichment、graph merge、citation-fixer、contradiction probe、dream cycle 或其他复杂后处理。
- 预处理能力按文件类型和业务语义拆成独立 Skill / 脚本，集中放在 `backend/skills/preprocessors/`，并通过中文 README 说明输入、输出、不做什么、模型路由、验收样本和新增 Skill 方法。

## 项目质量测试样本 (Project Quality Test Sample)

项目质量测试样本是用于验证项目资料录入、文件预览、引用定位和项目 `/query` 质量的真实业务文件集合，不要求其中所有文件来自同一个业务项目。

- 项目质量测试样本可以包含真实项目资料；项目编号或项目名称不一致本身不构成回归失败。
- 项目质量测试样本的验收重点是文件类型覆盖、预处理结果、GBrain source 隔离、答案命中、来源文件定位、页码/区域/时间戳等 citation 质量。
- 当同一事实存在多种文件载体时，项目质量测试样本应按文件载体分别建立测试问题，分别验证 PDF、图片、Office、音视频、邮件等来源链路。
- 项目质量测试样本中的期望定位可采用宽容结构，用于表达页码、sheet、图片区域或时间戳等定位意图；人工预估值不应直接作为严格失败条件。
- 项目质量测试样本中的答案断言应区分必须命中的核心词和宽容候选词，避免模型自然语言表达差异造成误失败。
- 项目质量测试样本允许执行真实预处理、生成 source-scoped 过程产物、写入 GBrain-ready Markdown 并触发当前测试项目 source sync；该许可不外溢到其他真实项目目录。
- 人工编写的问题草稿可保留在项目测试目录中；机器回归应使用独立 fixture，避免把草稿格式错误误判为产品失败。
- 项目质量测试样本的机器 fixture 应在顶层记录样本路径、是否使用真实项目文件、是否要求同一项目身份一致等范围信息。
- 项目质量测试样本的每条机器回归用例应显式标注文件类型，以便分别验收不同 extractor、预览和 citation 链路。
- 尚未实现但属于项目质量目标的能力可以进入机器 fixture，并用已知缺口状态标记；不得把 known gap 伪装成已通过。
- 已知缺口用例不计入失败通过率，但必须在质量报告中单独列出；当已知缺口实际通过时，应提示可升级为正式通过用例。
- 项目质量测试样本不是公司全局知识库样本，也不是可沉淀知识候选稿。

## Skill

Skill 是公司业务工作流的可执行封装。

- 内置 Skill 存放在 `backend/skills/builtin/`。
- 资料预处理 Skill 存放在 `backend/skills/preprocessors/`，它们不是普通业务流程 Skill，而是把源文件转换为 GBrain 友好 Markdown 的独立预处理能力。
- 标签打印源文件生成已退役，不再作为 Project_R 的 Skill 样板、测试目标或业务能力示例。
- Chat 负责发现意图和澄清，Agent 更积极承接执行、补参、生成文件和读取项目上下文。
- 修改 Skill 时同步更新根目录 `Project_R 业务工作流清单.md`。

## 文件生成 (Generated File)

文件生成是由 Chat 或 Skill 触发的受控输出能力。

- 已完成 Word tracer bullet；早期 U03 标签打印 Excel tracer bullet 不再作为产品验收目标。
- 生成文件必须通过后端权限校验下载。
- 运行时生成物不入 Git，属于本机运行数据。

## 通知事件 (Notification Event)

通知事件是 Project_R 内部发生且需要异步告知用户的业务事实。

- 典型事件包括 Skill 完成、Skill 阻断、项目资料索引完成、权限变更、知识审核待处理和系统健康告警。
- 通知事件本身不等于用户收件箱里的通知。
- 第一版不为通知事件建立独立事件表，可通过同批通知记录上的事件标识进行弱关联。

## 通知记录 (Notification Record)

通知记录是某个具体用户在通知中心看到的一条可读/未读消息。

- 每条通知记录只属于一个接收用户。
- 系统广播不使用 `ALL` 虚拟接收人，而是把同一个通知事件分发为多条通知记录。
- 用户的已读、未读和后续免打扰状态以通知记录为准。
- 通知记录默认不是长期审计证据；清理通知记录不得删除审计日志或原始业务记录。
- 通知中心入口位于右上角全局系统区，是当前用户的全局通知入口。
- 通知中心默认展示当前用户全部通知记录，并支持按当前工作区筛选。
- 通知中心只承载通知列表、状态筛选和白名单动作跳转，不承载完整业务页面。

## 通知分类 (Notification Category)

通知分类描述通知事件来自哪类业务场景。

- 第一版分类为：系统、任务、项目协同、审批、风险。
- 分类用于通知中心筛选，不表达紧急程度。

## 通知级别 (Notification Severity)

通知级别描述通知记录对接收人的处理紧急程度。

- 第一版级别为：信息、成功、警告、严重。
- 是否强提醒、是否需要用户处理和是否只收纳进通知中心，以通知级别和业务动作共同判断。
- 第一版强提醒只面向新产生的任务结果、任务阻断和严重风险；历史未读和普通协作信息只收纳进通知中心。

## 待办型通知 (Actionable Notification)

待办型通知是需要接收人完成业务动作才能闭环的通知记录。

- 已读只表示用户看过通知，不表示业务已处理。
- 待办状态用于表达业务是否已闭环，第一版状态为：无需处理、待处理、已完成、已忽略。
- 知识审核、人工介入请求、异常审计确认和商业风险告警属于待办型通知。
- 任务完成、系统更新和普通项目资料更新默认不是待办型通知。
- 第一版不提供按分类或按项目关闭通知；关键风险通知不能被普通偏好静音。

## 通知动作 (Notification Action)

通知动作是用户点击通知后由前端按白名单执行的结构化操作。

- 新通知优先使用动作类型和动作参数表达跳转或处理目标。
- 旧通知链接只作为兼容字段，不作为长期主要交互协议。
- 第一版动作类型聚焦打开会话、打开项目、打开 Skill 运行、下载文件和打开管理员审核。

## 通知分发 (Notification Delivery)

通知分发是把一个通知事件转换为一个或多个用户通知记录的过程。

- 普通项目文件变化默认进入工作区动态，不进入全局通知中心。
- 项目资料索引完成、索引失败、权限变更、管理员删除他人文件和批量删除异常可触发通知分发。
- 项目风险类资料更新第一版不自动识别内容，可由后续 Agent 或管理员触发通知。
- 第一版通知触发源聚焦 Skill/文件生成结果、知识审核、工作区权限与索引、系统风险告警。

## 工作区动态 (Workspace Activity)

工作区动态是项目资料容器内普通协作流水的轻量记录。

- 普通上传、新建文件夹和一般文件变更属于工作区动态。
- 工作区动态用于项目文件面板内查看，不默认推送到全局通知中心。

## 广播通知 (Broadcast Notification)

广播通知是同一个通知事件面向一组用户分发后的多条通知记录。

- 接收组可以是全员、管理员、项目成员、Skill 发起人或其他明确用户集合。
- 前端和普通通知查询只处理当前用户自己的通知记录，不读取共享广播对象。

## 临时内网测试部署 (Temporary LAN Test Deployment)

临时内网测试部署是在正式 Mac mini 后端到位前，由一台开发 Windows 电脑监听内网地址并给同事客户端测试使用的过渡部署形态。

- 临时内网测试部署不改变长期 Mac mini 后端部署目标。
- 员工客户端仍通过配置读取后端地址，不在请求层写死 `localhost` 或具体 IP。
- 测试安装包可以预置当轮可用的内网后端地址，但用户仍可在首次引导或设置页修改。

## 客户端更新 (Client Update)

客户端更新是员工桌面端从公司内网后端获取新版本安装包并完成升级的交付流程。

- 更新通知只负责提醒用户存在新版本，不负责直接替换程序文件。
- 安装包由 Project_R 后端在公司内网分发，不默认依赖 GitHub Releases。
- 更新过程必须由 Electron 主进程或安装器执行，不能由 renderer 页面直接覆盖应用文件。
- 客户端更新分为普通更新和强制更新；低于最低支持版本的客户端必须更新后才能继续使用。
- 更新交互先展示版本号与更新日志，再进入下载进度窗口，下载完成后提示用户重启完成安装。
- 更新包下载必须在用户确认后开始，不在登录后自动后台下载。
- 更新失败提示使用公司内部语境，引导用户联系管理员获取最新版安装包。
- 更新可用性按当前客户端版本本地检查，不按用户通知记录判断；通知中心只承载更新提醒入口。
- 第一版使用 `electron-builder` 生成安装包，Project_R 自行实现版本检查、下载安装包、SHA256 校验和启动安装器，不默认采用 `electron-updater` 全套发布机制。
- 版本检查接口可在内网未登录访问；安装包下载需要登录鉴权；更新包发布或登记仅限系统管理员。
- 强制更新允许用户完成登录以取得下载权限，但会在进入主工作台前阻断；用户只能下载并安装或退出软件。
