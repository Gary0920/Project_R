## 0. 为什么需要本文件

AI agent 改前端反复翻车（莫名卡片、颜色混乱、字号混乱、排版混乱、风格错误、僵硬无动效、图标位置/尺寸错误）的根因只有一个：**每次改动都在凭感觉重新发明一套视觉，而不是从唯一真相源取值。**

本文件的目标是把 Gemini 提供的视觉升级思路整理成可执行契约：

- 所有颜色、圆角、层级、间距、字号、图标尺寸、动效都有既定取值，必须复用，禁止硬编码。
- 普通 active / 当前项 / 轻量开启态统一使用“碧落暮蓝”控制色，不再使用绿色作为默认选中态。
- 卡片、动效、图标只在规定场景使用。
- 可视化规格书只做“全景图表字典”，不承担产品 Demo 或动态聊天模拟职责。

真相源代码位置仍以 [`frontend/src/renderer/shared/styles/base.css`](../../frontend/src/renderer/shared/styles/base.css) 为落地目标；若采纳本稿，需要同步更新 `base.css` 中对应 token。

## 1. 如何使用本文件

1. **取值优先级**：CSS 变量（碧落暮蓝 token）> 本文件量化阶梯 > 现有同类组件实测值。三者都没有时，才允许新增，且必须先更新本文件，再改 CSS。
2. **严禁硬编码**：不得在组件里写死 `#xxxxxx` 颜色、随手的 `z-index: 9999`、临时的 `border-radius` 或一次性字号。一律用 `hsl(var(--token))` 或本文件约定阶梯。
3. **浅色精细度优先**：核心用户长期处于默认浅色办公环境，浅色模式的明度差、边框、背景、hover、active 需要精细到 1%~5% 的层次差，消除死白和刺眼纯黑。
4. **改前先搜**：动手前先用搜索找现有类名、token 和相似组件，能复用就复用，不要并行造第二套。
5. **一个任务一个目标**：禁止“顺手优化”无关视觉。
6. **本文件是候选活文档**：采纳本稿后，新增视觉规则需要同步更新本文件、可视化规格书和 `base.css`。

## 2. 设计哲学与关键词

Project_R 是公司内部 AI 智能办公工具，也是面向幕墙、门窗、开模、顾问沟通、项目资料和客户情报场景的 Agent 工作台。它不是营销站点，也不是单纯问答机器人，而是能理解任务、拆解步骤、调用工具、处理文件、展示过程并交付结果的智能工作台。

长期 UI 原则：

- **Project_R Workbench first**：以企业级 Agent 工作台为视觉与交互基线；历史 Proma 只作阶段参考，不再是强约束。
- **Project_R core always**：业务语义、权限、文件、知识库、Skill、审计、执行链路服从后端架构。
- **Codex-style execution**：Agent 任务必须透明、可追踪、可确认、可复盘。
- **Biluo clarity**：用低饱和暮蓝灰阶替代死白、纯黑和临时高饱和色，提供更适合长时间办公的视觉稳定性。

设计关键词：**专业、克制、高效、透明、可控、通透**。

气质底线：

- 视觉重点是透明的执行过程与高可信度的结果交付，而不是炫酷。
- 控件状态一眼可辨，但不抢聊天内容、项目资料、图纸引用和执行结果的视觉焦点。
- 不使用装饰性大色块、渐变球、过度动画或营销式 hero。
- 高亮不靠“亮”和“艳”，而靠结构、明度差、细边框和轻微物理反馈。

## 3. Token 契约（碧落暮蓝动态调色流）

> 颜色变量采用 **HSL 通道格式**（候选种子色参考 `#AFD0F2` 动态衍生）。使用时必须包一层 `hsl()`：`background: hsl(var(--background));`。带透明度时用 `hsl(var(--token) / 0.5)`。

### 3.1 颜色系统

**基础语义变量（浅色候选映射）：**

| Token | 候选 HSL | 用途 |
|---|---:|---|
| `--background` | `212 45% 98%` | 页面大背景：极淡暮蓝避光白，消除死白 |
| `--foreground` | `212 40% 12%` | 主文字：深暮蓝，代替纯黑，降低锐利感 |
| `--content-area` | `212 30% 99%` | 核心内容/对话流底色 |
| `--card` / `--card-foreground` | `0 0% 100%` / `212 40% 12%` | 卡片、侧栏、输入框底色与文字 |
| `--muted` | `212 25% 94%` | 次级背景、未激活项、轻 hover |
| `--muted-foreground` | `212 15% 48%` | 占位符、时间戳、次级 meta |
| `--border` | `212 24% 90%` | 默认分区线与控件细边框 |
| `--primary` / `--primary-foreground` | `212 40% 12%` / `0 0% 98%` | 主按钮/发送按钮（深色实心） |
| `--secondary` / `--accent` | `212 25% 94%` | 次级背景块 |

**状态语义色（仅在语义明确时使用）：**

| Token | 语义 | 允许场景 |
|---|---|---|
| `--success` | 完成、成功、测试通过 | 完成态勾号、success tag |
| `--warning` | 等待、注意、需要确认 | 待确认提示、本机待授权附件 |
| `--destructive` | 删除、失败、危险、强警告 | 删除、失败态、危险确认 |

**选中态（普通 active / 当前项 / 激活态的唯一配色）—— 碧落暮蓝：**

```text
--control-active-bg         212 55% 93%  /* 碧落淡蓝选中底 */
--control-active-bg-hover   212 58% 95%  /* 选中态 hover */
--control-active-border     212 45% 85%  /* 激活态细边框 */
--control-active-foreground 212 60% 32%  /* 碧落深蓝选中文字 */
--control-active-shadow     0 6px 16px -4px rgb(175 208 242 / 0.25)
--brand-accent              212 55% 40%  /* 核心高亮与引导轴色 */
```

适用：当前工作区/会话/标签/列表项、输入框功能按钮开启态（深度思考、联网、提示词、技能）、Skill 候选 hover/active、轻量功能入口 hover、拖拽目标 hover。

**能力提示色：**

蓝色不再作为普通 active 的临时色；若需要标识 `/query`、企业 Skill、提示词等“已选能力提示”，优先使用 `--capability-*` 或 `--control-active-*` 的碧落体系，不再引入独立亮蓝。

**颜色红线：**

- 禁止为普通 active 临时引入亮蓝、紫色、橙色、深黑或任何新色；普通选中态只有碧落暮蓝一种逻辑。
- 禁止同一工具栏内出现多套 active 配色逻辑。
- 主按钮使用 `--primary`，但普通 toggle active 不使用主按钮黑底。
- 深色主题必须保留同一语义，不得把浅色卡片直接嵌入深色界面。
- 禁止硬编码 hex/rgb；只能用上述 token。大面积高饱和色一律禁止。

### 3.2 字号阶梯（Type Scale）

实测代码字号集中在 10–26px。统一收敛为下表，禁止在卡片、侧栏、工具栏、设置项标题使用 hero 级字号。

| 层级 | 字号 | 用途 |
|---|---|---|
| 微标 micro | `10–11px` | 角标、计数、极次要 meta |
| 辅助 caption | `12px` | 状态标签、次级 meta、时间戳 |
| 正文小 body-sm | `13px` | 控件文字、列表项、按钮文字、密集正文 |
| 正文 body | `14px` | 默认正文、聊天正文、表单 |
| 副标题 subtitle | `15–16px` | 区块标题、面板标题、强调正文 |
| 标题 title | `17–20px` | 页面/弹窗主标题 |
| 大标题 display | `21–26px` | 仅限欢迎/空状态等极少数主屏，业务工作区禁用 |

排版规则：

- 正文 `14px`（密集场景可 `13px`）；代码、路径、命令行输出、材料编号用 `13–14px` 等宽字体。
- 标题清晰但不过大；不要用 hero 字号处理卡片、侧栏、工具栏、设置项标题。
- 长内容默认折叠，需要时展开。
- 行高保持稳定，不逐组件临场自定义。

字体分工：

| 内容类型 | 字体 |
|---|---|
| 普通文字 | 系统默认（`Microsoft YaHei UI` / `Segoe UI` / system-ui，组件不要另设 font-family） |
| 代码 / 路径 / 命令行 / 材料编码 | 等宽字体（JetBrains Mono / Fira Code / Consolas） |

### 3.3 间距阶梯（Spacing Scale）

新代码统一使用 4 的倍数阶梯，禁止再引入奇值：

| 档位 | 值 | 典型用途 |
|---|---|---|
| 紧凑 | `2px` / `4px` | 图标与文字、标签内边距、密集列表行内 |
| 标准 | `6px` / `8px` | 控件内 gap、按钮内边距、列表项间距（`--panel-gap: 8px`） |
| 区块 | `12px` / `16px` | 卡片内边距、区块间距、面板内边距 |
| 大区 | `24px` | 大分区之间、空状态留白 |

- 卡片之间、区块之间靠间距和细边框分区，而不是层层套卡片。
- 壳层内边距用既有 `--shell-pad: 8px`。
- 顶栏高度用 `--workbench-topbar-height: 42px` / `--app-topbar-height: 36px` / `--app-section-header-height: 48px`。

### 3.4 圆角阶梯（Radius Scale）

| Token | 值 | 用途 |
|---|---|---|
| `--shell-radius` | `12px` | 壳层、大面板外框 |
| `--card-radius` | `12px` | 消息、计划、弹窗、对象卡片 |
| `--control-radius` | `9px` | 按钮、输入框、标签。基础 button 可保持 8px，新控件向 9px 靠拢 |

### 3.5 图标尺寸阶梯（Icon Size Scale）

| 档位 | 尺寸 | 配套场景 |
|---|---|---|
| 微 | `12–13px` | 12px 辅助文字旁的 meta 图标、角标 |
| 标准 | `14px` | 列表项、侧栏项、菜单项、随 13–14px 文字 |
| 控件 | `15–16px` | 工具栏按钮、输入框功能按钮、发送按钮内图标 |
| 强调 | `18px` | 区块标题旁图标、空状态主图标 |

图标规则：

- 同一组工具栏/同一类列表里图标尺寸必须统一。
- 图标必须 `flex-shrink: 0`，文字截断时图标不变形。
- 图标只表达功能，不做装饰。
- 容器变窄时只隐藏显式 label，保留图标且尺寸不变。

### 3.6 层级 z-index（Layer Scale）

只能用命名 token，禁止魔法数字：

| Token | 值 | 用途 |
|---|---:|---|
| `--layer-popover` | `90` | 浮层、下拉、tooltip |
| `--layer-toast` | `95` | toast 通知 |
| `--layer-context-menu-backdrop` | `96` | 右键菜单遮罩 |
| `--layer-context-menu` | `97` | 右键菜单 |
| `--layer-modal` | `100` | 普通弹窗 |
| `--layer-critical-modal` | `110` | 关键确认弹窗 |

### 3.7 边框、阴影、毛玻璃与卡片

业务工具界面优先用细边框 + 留白分区，边框用 `--border`。阴影只用于浮层、弹窗、输入框容器和 active 工具按钮的轻微反馈；普通区块不加重阴影。

候选物理材质 token：

```text
--shadow-sm     0 1px 2px 0 rgb(175 208 242 / 0.10)
--shadow-md     0 6px 16px -4px rgb(175 208 242 / 0.25), 0 2px 6px -2px rgb(175 208 242 / 0.15)
--glass-bg      rgb(255 255 255 / 0.82)
--glass-border  rgb(175 208 242 / 0.35)
--glass-blur    blur(14px)
```

适用：临时浮层、通知气泡、Popover、检索建议、短生命周期提示。不得把毛玻璃用于主工作区、聊天正文、文件列表主体。

卡片只用于具体对象：消息、计划、工具调用、文件变更、结果、Review 项、弹窗、真正需要框住的工具。不要把页面 section 做成层层嵌套的浮动卡片。

## 4. 动效契约

动效只用于帮助用户理解状态变化，不做装饰。基础时长仍保持原有阶梯，但 Gemini 版本增加“物理反馈”候选规则。

| 档位 | 时长 | 缓动 | 用途 |
|---|---:|---|---|
| 微交互 | `0.12–0.16s` | `ease` / `var(--spring-curve)` | hover、active、颜色/边框/背景过渡、按钮按下 |
| 面板/卡片 | `0.18–0.28s` | `ease-out` / `var(--spring-curve)` | 卡片展开、抽屉滑入、宽度变化 |
| 进入/退出 | `0.25–0.34s` | `ease-in-out` / `ease-out` | toast、drawer 进出场、流式消息淡入 |
| loading | `0.8–1s` | `linear` | spinner、blink，匀速循环 |

候选物理曲线：

```text
--spring-curve: cubic-bezier(0.34, 1.56, 0.64, 1)
--transition-micro: all 0.16s var(--spring-curve)
--transition-panel: all 0.28s var(--spring-curve)
```

1. **阻尼物理曲线**：右侧上下文面板、大抽屉、来源预览面板滑出时可使用 `0.28s cubic-bezier(0.34, 1.56, 0.64, 1)`，形成轻微终点回弹。仅限大面板，不用于每个小按钮。
2. **按压形变触感**：可点击的独立按钮、会话卡片、Pill 标签 hover 时可轻微上浮 `translateY(-1px)` 到 `translateY(-2px)`；active 时可使用 `scale(0.96)` 或 `translateY(1px)`，但不得造成布局位移。
3. **流式淡入**：AI 气泡、执行步骤出现时可从 `translateY(8px)` 到 `0`，`opacity: 0` 到 `1`，时长约 `0.34s`，减少机械闪烁。

允许：Agent 执行中的轻微 loading；步骤完成的轻微状态变化；卡片展开平滑过渡；新结果轻微淡入；错误轻微高亮。

禁止：大面积动画；强烈闪烁；复杂转场；装饰性动效堆叠；高饱和多色 loading；英文随机词/拟人俏皮 loading 文案。

无障碍：必须尊重 `prefers-reduced-motion`，用户开启减弱动效时关闭非必要动画。

## 5. 布局与信息层级

### 5.1 三栏工作台（与代码对齐）

| 栏 | 职责 |
|---|---|
| 左：导航区 | 品牌、Agent/Chat 模式切换、工作区切换（`个人` / `项目` / `CRM` 三组）、新建对话、会话列表、左下角当前用户 + **设置、登出**。实现于 [`ChatSidebar.tsx`](frontend/src/renderer/features/chat/components/ChatSidebar.tsx)。 |
| 中：执行区 | **单行标题栏**（CRM / 便签 / Tab / 通知 / 窗口控制，Chrome 式合并）+ 对话/Agent 执行流。**双对话**在此区左右并排（`.chat-workbench.is-split`），不在右栏。 |
| 右：上下文预览区 | **单一活动** `utilityPanel`：文件管理、CRM 工作台、客户情报、提示词、Skills、来源预览等。GBrain 录入/状态在文件面板或管理后台内，无独立 `knowledge` 壳层面板。 |

导航位置约定（与 `AppWorkspaceChrome.tsx` 一致）：

- **标题栏（单行 `.workbench-titlebar`）**：CRM（客户工作区）；便签 Scratch；会话 Tab（**当前 Tab 前归属小标签**）；右侧 **通知** + 窗口控制
- **聊天标题栏**：置顶、**双对话**、**文件管理**（项目/客户；个人工作台隐藏）、**客户情报**（客户工作区）
- **Composer 底栏**：纸夹附件、模型、深度思考、联网、提示词、技能、发送
- **左下角用户区**：当前用户身份 + **设置、登出**，不在顶部重复头像
- **侧栏视觉（碧落 v3.5）**：区块用 `8px/12px` gap，禁止多层横线；仅用户区顶部分隔线；品牌行右端为「搜索对话 + 收起」；新建对话独立全宽主按钮；会话 active 用 `--control-active-*`；分组标签 11–12px muted
- **工作区**：标题行无 SearchIcon；管理员「浏览项目目录」在工作区列表底部
- **Scratch 便签**：Tab 栏 `.tab-note-btn` 占满中间执行区，非右侧面板

### 5.2 信息四层（默认只展示第一、二层）

| 层 | 内容 | 默认 |
|---|---|---|
| 第一层 | 任务是否完成、Agent 在做什么、是否需确认、最终结果 | 展示 |
| 第二层 | 执行计划、当前步骤、工具调用摘要、文件修改摘要 | 展示 |
| 第三层 | 命令行输出、代码 diff、错误日志、测试详情 | 折叠/展开 |
| 第四层 | 全部工具调用、中间输出、完整上下文、版本记录 | 详情页/Review |

辅助信息默认用低对比信息条或轻量面板，不得比用户消息、Agent 计划、执行结果更抢焦点。

## 6. 组件规则

### 6.1 状态模型（统一用词，禁止逐组件自造状态词）

**Thread 状态：** `Draft` / `Planning` / `Running` / `Waiting Approval` / `Need Review` / `Failed` / `Completed` / `Archived`。

**Step / Tool 状态显示：** 等待中=灰点/muted 文案；执行中=轻量 loading；已完成=绿色勾/success tag；需确认=黄色提示；出错=红色警告。状态标签必须小尺寸、浅背景、低饱和，不抢正文焦点。

**录入状态用词：** `queued` / `preprocessing` / `gbrain_ready` / `sync_pending` / `synced` / `failed` / `pending_capability` / `ignored` / `source_changed` / `needs_repreprocess`。

### 6.2 Agent 执行组件（新增执行 UI 一律映射到这些模式，不要另造）

任务理解卡片、计划卡片、执行步骤卡片/时间线、工具调用卡片、文件变更卡片、结果卡片、Review 面板。计划卡片要短、可扫读，不写成长文档。

### 6.3 普通 Chat 过程状态

等待模型返回时保留加载动画 + 轻量过程状态（“正在理解问题”“正在整理上下文”“正在生成回答”“正在处理本轮附件”）。不显示 Agent 计划控件；未显式 `/query` 时不得显示“正在查询 GBrain / 读取项目·客户 source”。

### 6.4 输入框

- 底部功能按钮是一组同级工具，不按功能来源拆成不同视觉系统。高度 `32px`、圆角 `8px` 或 `--control-radius`、默认细边框、active 统一用 `--control-active-*`。
- 响应式优先按容器宽度判断，不只看 viewport。
- 附件预览每个文件独立卡片；图片显示真实缩略图；PDF/通用文件显示文件卡片；meta 拆成短标签可换行。
- 已选能力提示（`/query`、公司预设 Skill、提示词）用统一轻量卡片：左能力图标、中能力名、右小号分类（`提示词` / `公司预设` / `内置命令`），必须有取消入口。

### 6.5 列表项与侧栏项

当前项用碧落淡蓝填充，不加重边框；hover 用更淡的同色；图标底色可用同一体系的浅背景但不用高饱和渐变；状态标签放次级位置，不挤压标题。

### 6.6 文件面板（右侧常驻，仅项目/客户资料；个人工作台无个人文件面板）

- 默认紧凑列表：每行小图标 + 文件名 + 大小/入库状态。
- 不展示底层目录（`.git`、`derived`、`manifests`、`.pending_review`、`_preprocessed`、`gbrain-ready`、`runs` 等后端产物）。
- 预览不插入列表内部；点击文件在面板侧面打开可关闭的预览窗。
- 路径栏用 Windows 地址栏逻辑：后退/前进/上一级 + `›` 分隔层级。
- 颜色只复用 `--control-active-*` / `--success` / `--warning` / `--destructive` / `--muted` / `--border` / `--card`。
- 录入二次确认必须说明：递归处理、预计文件数、主要类型、高成本 MiMo/PDF/转写风险、当前权限。

### 6.7 确认弹窗与用户控制

关键操作（改文件、删除、覆盖、跑高风险命令、提交代码、外部请求、批处理、生成/替换正式文件）前必须确认。弹窗说明要做什么、影响范围、风险、可选操作。主操作按钮只用于明确确认；取消/修改/查看详情为次级。

## 7. 反模式画廊（Gary 的痛点 → 正确做法）

| 痛点 | 错误 | 正确 |
|---|---|---|
| 莫名卡片 | 把每个页面区块都做成带边框+阴影的浮动卡片，层层嵌套 | 卡片只用于消息/计划/工具调用/文件变更/结果/Review/弹窗；大区块用分区+细边框+留白 |
| 颜色混乱 | 随手写 `#3b82f6`、给 active 用蓝/紫/橙、深浅主题各写一套硬编码色 | 一律 `hsl(var(--token))`；普通选中态只用碧落 `--control-active-*` |
| 字号混乱 | 同类标题忽 14 忽 22；卡片标题用 hero 字号 | 按字号阶梯取值；卡片/侧栏/工具栏禁用 hero |
| 排版混乱 | gap/padding 随手 7/9/13/18 乱填 | 用 4 的倍数间距阶梯；复用 `--panel-gap` 等 token |
| 圆角混乱 | 每个组件一个随机圆角 4/5/6/10 | 用 `--shell-radius` / `--card-radius` / `--control-radius` |
| 图标乱 | 相邻按钮 14 与 18 混用；图标随文字截断变形 | 同组统一尺寸、`flex-shrink: 0` |
| 僵硬无动效 | 交互态直接硬切，无过渡 | hover/active/选中/展开加微交互过渡 |
| 动效过度 | 大面积动画、闪烁、高饱和多色 loading | 只用帮助理解状态的轻动效 |
| z-index 大战 | 到处 `z-index: 9999` | 用 `--layer-*` 命名层级 |
| 深色主题破相 | 把浅色卡片直接塞进深色界面 | 深色用深色面板底 + 状态色边框，同一语义换深色 token |

## 8. Agent 交付前自检清单（每次前端改动必须逐条核对）

- [ ] 改动前已读本文件，并搜索过现有类名/token/相似组件，能复用的已复用。
- [ ] 页面无任何硬编码 hex/rgb 颜色，全部走 `hsl(var(--token))` 碧落暮蓝体系。
- [ ] 普通激活项底色为碧落淡蓝 `--control-active-bg`，文字为暮蓝深色 `--control-active-foreground`。
- [ ] 普通 active 没有新引入蓝/紫/橙；主按钮仍使用 `--primary`。
- [ ] 常规按钮、Pill、会话项按下时具备合理的微反馈（如 `scale(0.96)` 或 `translateY(1px)`），且不造成布局位移。
- [ ] 右侧上下文/来源预览面板滑出可使用 `--spring-curve`，但小控件不过度弹簧化。
- [ ] 字号取自字号阶梯，卡片/侧栏/工具栏/设置项标题未用 hero 字号。
- [ ] 间距用 4 的倍数阶梯（或既有 token），无随手奇值；圆角用 `--*-radius`。
- [ ] 图标尺寸取自图标阶梯，同组统一，`flex-shrink: 0`，位置符合布局职责。
- [ ] z-index 用 `--layer-*`，无魔法数字。
- [ ] 新卡片确实属于具体对象场景，没有把普通 section 做成浮动卡片或层层嵌套。
- [ ] 交互态（hover/active/选中/展开）有过渡，不僵硬；无大面积/闪烁/装饰动效；尊重 `prefers-reduced-motion`。
- [ ] 深浅主题都验证过，深色未塞浅色卡片。
- [ ] 容器变窄时按钮自动收为图标模式，标签隐藏但图标尺寸/逻辑不变。
- [ ] 新增视觉规则已先更新本文件和可视化规格书，再改 CSS。

## 9. 修改流程与验证门槛

1. 先搜索现有类名、token、相似组件，判断改动属于壳层视觉、业务语义还是 Agent 执行流体验。
2. 新增同类按钮/列表项/卡片/状态，必须复用现有 token；新增 Agent 执行组件，映射到 §6.2 既有模式。
3. 确需新视觉规则：先更新本文件，再改 CSS。
4. 逐条过 §8 自检清单。
5. 完成后在 `frontend` 运行 `bun run typecheck`；涉及打包、路由或构建配置补跑 `bun run build`。
6. 涉及可见 UI 的改动，用浏览器或 Electron 做一次截图 / DOM 验证（深浅主题各一次）。
