# Project_R UI 设计语言与设计约束（唯一真相源 · 碧落暮蓝 v3.5）

> 本文件是 Project_R 前端视觉的**唯一权威设计契约**。任何前端视觉改动（新增/修改组件、样式、布局、颜色、字号、图标、动效）开始前**必须先读本文件**，结束前**必须逐条核对第 8 节《Agent 交付前自检清单》**。
>
> **2026-06-22 视觉升级**：普通选中态已从「克制绿」迁移为「碧落暮蓝」token 体系，已落地于 [`base.css`](../../frontend/src/renderer/shared/styles/base.css)。完整候选稿与可视化规格见 [`ui-design-language-Gemini.md`](./ui-design-language-Gemini.md) 与 [`ui-design-language-visual-Gemini.html`](./ui-design-language-visual-Gemini.html)。
>
> 本文件取代旧版 `docs/design/ss/ui-design-language.md`，并吸收其全部仍有效的产品/交互语义。旧文件已废弃。

## 0. 为什么需要本文件

AI agent 改前端反复翻车（莫名卡片、颜色混乱、字号混乱、排版混乱、风格错误、僵硬无动效、图标位置/尺寸错误）的根因只有一个：**每次改动都在凭感觉重新发明一套视觉，而不是从唯一真相源取值。**

本文件的目标就是消灭"临场发明"：

- 所有颜色、圆角、层级、间距、字号、图标尺寸、动效都有**既定取值**，必须复用，禁止硬编码。
- 卡片、动效、图标只在**规定场景**使用。
- 每次交付前用统一 checklist 自检。

真相源代码位置：[`frontend/src/renderer/shared/styles/base.css`](../../frontend/src/renderer/shared/styles/base.css)（`:root` / `:root[data-theme="dark"]` 中定义全部 CSS 变量）。

## 1. 如何使用本文件

1. **取值优先级**：CSS 变量（token）> 本文件量化阶梯 > 现有同类组件的实测值。三者都没有时，才允许新增，且**必须先更新本文件，再改 CSS**。
2. **严禁硬编码**：不得在组件里写死 `#xxxxxx` 颜色、随手的 `z-index: 9999`、临时的 `border-radius` 或一次性字号。一律用 `hsl(var(--token))` 或本文件约定阶梯。
3. **改前先搜**：动手前先用搜索找现有类名、token 和相似组件，能复用就复用，不要并行造第二套。
4. **一个任务一个目标**：禁止"顺手优化"无关视觉。
5. **本文件是活文档**：确实需要新视觉规则时，先在本文件登记规则与 token，再落 CSS。

## 2. 设计哲学与关键词

Project_R 是公司内部 AI 智能办公工具，也是面向企业业务场景的 Agent 工作台。它不是营销站点，也不是单纯问答机器人，而是能理解任务、拆解步骤、调用工具、处理文件、展示过程并交付结果的智能工作台。

长期 UI 原则：

- **Project_R Workbench first**：以企业级 Agent 工作台为视觉与交互基线；历史 Proma 只作阶段参考，不再是强约束。
- **Project_R core always**：业务语义、权限、文件、知识库、Skill、审计、执行链路服从后端架构。
- **Codex-style execution**：Agent 任务必须透明、可追踪、可确认、可复盘。

设计关键词：**专业、克制、高效、透明、可控**。工具化、低干扰、面向执行结果；信息密度适中，适合长时间工作。

气质底线：

- 视觉重点不是炫酷，而是让用户清楚知道 Agent 理解了什么、正在做什么、用到了什么、改了什么、哪里要确认、交付物在哪。
- 不使用装饰性大色块、渐变球、过度动画或营销式 hero。
- 控件状态一眼可辨，但不抢聊天内容、项目资料和执行结果的视觉焦点。

## 3. Token 契约（核心）

> 下表取值与 `base.css` 实际定义一致。颜色变量采用 **HSL 通道格式**（如 `--background: 0 0% 100%`），使用时必须包一层 `hsl()`：`background: hsl(var(--background));`。带透明度时用 `hsl(var(--token) / 0.5)`。

### 3.1 颜色系统

**基础语义变量（亮/暗主题各有一套，自动切换，组件不要写死颜色）：**

| Token | 用途 |
|---|---|
| `--background` / `--foreground` | 页面底色 / 主文字 |
| `--content-area` | 内容工作区底色 |
| `--card` / `--card-foreground` | 卡片底色 / 卡片文字 |
| `--muted` / `--muted-foreground` | 次级背景 / 次级文字（说明、占位、辅助信息） |
| `--border` | 默认边框 |
| `--primary` / `--primary-foreground` | 主按钮/发送按钮（深色实心），**不用于普通 toggle active** |
| `--secondary` / `--accent` | 次级背景块 |

**状态语义色（仅在语义明确时使用）：**

| Token | 语义 | 允许场景 |
|---|---|---|
| `--success` | 完成、成功、测试通过 | 完成态勾号、success tag |
| `--warning` | 等待、注意、需要确认 | 待确认提示、本机待授权附件 |
| `--destructive` | 删除、失败、危险、强警告 | 删除、失败态、危险确认 |

**选中态（普通 active / 当前项 / 激活态的唯一配色）—— 碧落暮蓝：**

```text
--control-active-bg        /* 碧落淡蓝选中底 */
--control-active-bg-hover  /* 选中态 hover */
--control-active-border    /* 选中边框 */
--control-active-foreground/* 选中文字 */
--control-active-shadow    /* 选中轻微阴影 */
--brand-accent             /* 核心高亮与引导轴色 */
```

适用：当前工作区/会话/标签/列表项、输入框功能按钮开启态（深度思考、联网、提示词、技能）、Skill 候选 hover/active、轻量功能入口 hover、拖拽目标 hover。

**能力提示（`/query`、公司预设 Skill、提示词）** 与 active 同属碧落体系（`--capability-*` 或 `--control-active-*`），不另造独立亮蓝。

**颜色红线：**

- ❌ 禁止为普通 active 临时引入亮蓝、紫色、深黑、橙色或任何新色；普通选中态只有碧落暮蓝一种逻辑。
- ❌ 同一工具栏内不得出现多套 active 配色逻辑。
- 主按钮使用 `--primary`，但普通 toggle active 不使用主按钮黑底。
- 深色主题必须保留同一语义，用暗色碧落变量；不得把浅色卡片直接嵌进深色界面。
- ❌ 禁止硬编码 hex/rgb；只能用上述 token。大面积高饱和色一律禁止。

### 3.2 字号阶梯（Type Scale）

实测代码字号集中在 10–26px。统一收敛为下表，**禁止在卡片/侧栏/工具栏/设置项标题使用 hero 级字号**。

| 层级 | 字号 | 用途 |
|---|---|---|
| 微标 micro | `10–11px` | 角标、计数、极次要 meta |
| 辅助 caption | `12px` | 状态标签、次级 meta、时间戳 |
| 正文小 body-sm | `13px` | 控件文字、列表项、按钮文字、密集正文 |
| 正文 body | `14px` | 默认正文、聊天正文、表单 |
| 副标题 subtitle | `15–16px` | 区块标题、面板标题、强调正文 |
| 标题 title | `17–20px` | 页面/弹窗主标题 |
| 大标题 display | `21–26px` | 仅限欢迎/空状态等极少数主屏，**业务工作区禁用** |

排版规则：

- 正文 `14px`（密集场景可 `13px`）；代码、路径、命令行输出 `13–14px` 并用等宽字体。
- 标题清晰但不过大；不要用 hero 字号处理卡片、侧栏、工具栏、设置项标题。
- 长内容默认折叠，需要时展开。
- 行高保持稳定，不要逐组件自定义行高。

字体分工：

| 内容类型 | 字体 |
|---|---|
| 普通文字 | 系统默认（`Microsoft YaHei UI` / `Segoe UI` / system-ui，已在 `base.css` 设定，组件不要另设 font-family） |
| 代码 / 路径 / 命令行 | 等宽字体（JetBrains Mono / Fira Code / Consolas） |

### 3.3 间距阶梯（Spacing Scale）

实测 gap/padding 较散（出现过 7/9/14/18 等奇值）。新代码统一用 **4 的倍数阶梯**，禁止再引入奇值：

| 档位 | 值 | 典型用途 |
|---|---|---|
| 紧凑 | `2px` / `4px` | 图标与文字、标签内边距、密集列表行内 |
| 标准 | `6px` / `8px` | 控件内 gap、按钮内边距、列表项间距（`--panel-gap: 8px`） |
| 区块 | `12px` / `16px` | 卡片内边距、区块间距、面板内边距 |
| 大区 | `24px` | 大分区之间、空状态留白 |

- 卡片之间、区块之间留足间距，靠间距和细边框分区，而不是层层套卡片。
- 壳层内边距用既有 `--shell-pad: 8px`；顶栏高度用 `--workbench-topbar-height: 42px` / `--app-topbar-height: 36px` / `--app-section-header-height: 48px`，不要写死新高度。

### 3.4 圆角阶梯（Radius Scale）

来自 `base.css`，直接复用变量：

| Token | 值 | 用途 |
|---|---|---|
| `--shell-radius` | `12px` | 壳层、大面板外框 |
| `--card-radius` | `12px` | 卡片 |
| `--control-radius` | `9px` | 控件（按钮、输入框、标签）。注意：`base.css` 中基础 `button` 当前用 `8px`，新控件统一向 `--control-radius` 靠拢，不要再引入 4/5/6/10 等随机圆角 |

### 3.5 图标尺寸阶梯（Icon Size Scale）

实测图标 10–18px，必须与所在控件/字号配比，**杜绝忽大忽小**：

| 档位 | 尺寸 | 配套场景 |
|---|---|---|
| 微 | `12–13px` | 12px 辅助文字旁的 meta 图标、角标 |
| 标准 | `14px` | 列表项、侧栏项、菜单项、随 13–14px 文字（最常用默认值） |
| 控件 | `15–16px` | 工具栏按钮、输入框功能按钮、发送按钮内图标（随 32px 控件） |
| 强调 | `18px` | 区块标题旁图标、空状态主图标 |

图标规则：

- 同一组工具栏/同一类列表里，图标尺寸必须统一；不允许相邻按钮一个 14 一个 18。
- 图标必须 `flex-shrink: 0`，文字截断时图标不变形。
- 图标只表达功能，不做装饰；功能图标位置遵循第 5 节布局职责（业务入口在顶部业务导航，系统动作在右上角全局区，当前用户固定左下角）。
- 业务功能入口用"图标 + 短文字"；系统工具入口可只用图标，但必须有 tooltip / 状态徽标 / 未读数。
- 容器变窄时只隐藏显式 label（如 `.composer-button-label`），保留图标且尺寸不变；不要写 `button span { display:none }`（图标也可能是 span）。

### 3.6 层级 z-index（Layer Scale）

只能用 `base.css` 命名 token，**禁止魔法数字（如 9999）**：

| Token | 值 | 用途 |
|---|---|---|
| `--layer-popover` | 90 | 浮层、下拉、tooltip |
| `--layer-toast` | 95 | toast 通知 |
| `--layer-context-menu-backdrop` | 96 | 右键菜单遮罩 |
| `--layer-context-menu` | 97 | 右键菜单 |
| `--layer-modal` | 100 | 普通弹窗 |
| `--layer-critical-modal` | 110 | 关键确认弹窗 |

### 3.7 边框、阴影与卡片

- 业务工具界面优先用**细边框 + 留白**分区，边框用 `--border`（深色主题透明度低，避免线框感过重）。
- 阴影只用于浮层、弹窗、输入框容器和 active 工具按钮的轻微反馈；普通区块不加阴影。
- 卡片只用于**具体对象**：消息、计划、工具调用、文件变更、结果、Review 项、弹窗、真正需要框住的工具。
- ❌ 不要把页面 section 做成层层嵌套的浮动卡片。页面大区块用工作区、分区、工具栏、抽屉、右侧上下文面板组织。

## 4. 动效契约

动效只用于**帮助用户理解状态变化**，不是装饰。实测过渡时长集中在 0.12–0.3s，统一收敛如下：

| 档位 | 时长 | 缓动 | 用途 |
|---|---|---|---|
| 微交互 | `0.12–0.16s` | `ease` | hover、active、颜色/边框/背景过渡、按钮按下 |
| 面板/卡片 | `0.18–0.24s` | `ease-out` | 卡片展开、抽屉滑入、宽度变化 |
| 进入/退出 | `0.25–0.3s` | `ease-in-out` / `ease-out` | toast、drawer 进出场 |
| loading | `0.8–1s` | `linear` | spinner、blink，匀速循环 |

**允许：** Agent 执行中的轻微 loading；步骤完成的轻微状态变化；卡片展开平滑过渡；新结果轻微淡入；错误轻微高亮。

**禁止：** 大面积动画；强烈闪烁；复杂转场；装饰性动效堆叠；高饱和多色 loading；英文随机词/拟人俏皮 loading 文案（loading 文案用稳定中文，如"正在理解问题""正在生成回答"）。

**无障碍：** 必须尊重 `prefers-reduced-motion`，用户开启减弱动效时关闭非必要动画。

**僵硬感的解法：** 交互态变化（hover/active/选中/展开）一律加 `0.12–0.16s` 过渡，不要直接硬切；这是消除"软件很僵硬"的最低要求。

## 5. 布局与信息层级

### 5.1 三栏工作台（与代码对齐）

| 栏 | 职责 |
|---|---|
| 左：导航区 | 品牌、Agent/Chat 模式、工作区（`个人` / `项目` / `CRM`）、新建对话、会话列表、左下角用户。当前项碧落淡蓝高亮。**设置仅在顶栏右上角**。 |
| 中：执行区 | 顶栏 + Tab（Scratch 便签、会话 Tab）+ 对话/Agent 流。**双对话**在此区并排，不在右栏。 |
| 右：上下文预览区 | 单一 `utilityPanel`：文件管理、CRM、客户情报、提示词、Skills、来源预览等。GBrain 在文件面板/管理后台内。 |

导航位置约定（功能图标不要乱放）：

- 顶栏左：归属路径；顶栏中（CRM 工作区）：CRM 业务按钮；顶栏右：通知、设置、登出
- 聊天标题栏：置顶、双对话、文件管理（个人工作台隐藏）、客户情报（客户工作区）
- 左下角用户区：当前用户身份，不在顶部重复头像
- 输入框：左侧一个纸夹 + 模型/深度思考/联网/创意度；右侧提示词/技能/发送

### 5.2 信息四层（默认只展示第一、二层）

| 层 | 内容 | 默认 |
|---|---|---|
| 第一层 | 任务是否完成、Agent 在做什么、是否需确认、最终结果 | 展示 |
| 第二层 | 执行计划、当前步骤、工具调用摘要、文件修改摘要 | 展示 |
| 第三层 | 命令行输出、代码 diff、错误日志、测试详情 | 折叠/展开 |
| 第四层 | 全部工具调用、中间输出、完整上下文、版本记录 | 详情页/Review |

辅助信息（本轮上下文、引用来源、通知摘要）默认用低对比信息条或轻量面板，不得比用户消息、Agent 计划、执行结果更抢焦点。

## 6. 组件规则

### 6.1 状态模型（统一用词，禁止逐组件自造状态词）

**Thread 状态：** `Draft` / `Planning` / `Running` / `Waiting Approval` / `Need Review` / `Failed` / `Completed` / `Archived`。

**Step / Tool 状态显示：** 等待中=灰点/muted 文案；执行中=轻量 loading（不闪烁）；已完成=绿色勾/success tag；需确认=黄色提示；出错=红色警告。状态标签必须小尺寸、浅背景、低饱和，不抢正文焦点。

**录入状态用词：** `queued` / `preprocessing` / `gbrain_ready` / `sync_pending` / `synced` / `failed` / `pending_capability` / `ignored` / `source_changed` / `needs_repreprocess`。文案要说明"源文件删除不等于知识删除"。

### 6.2 Agent 执行组件（新增执行 UI 一律映射到这些模式，不要另造）

任务理解卡片、计划卡片、执行步骤卡片/时间线、工具调用卡片（默认折叠、失败可自动展开错误）、文件变更卡片（含 diff 与接受/回退/重生成入口）、结果卡片（未验证必须明示"未验证"，不隐藏）、Review 面板。计划卡片要短、可扫读，不写成长文档。

### 6.3 普通 Chat 过程状态

等待模型返回时保留加载动画 + 轻量过程状态（"正在理解问题/正在整理上下文/正在生成回答/正在处理本轮附件"）。不显示 Agent 计划控件；未显式 `/query` 时不得显示"正在查询 GBrain / 读取项目·客户 source"。

### 6.4 输入框

- 底部功能按钮是一组**同级工具**，不按功能来源拆成不同视觉系统。高度 `32px`、圆角 `8px`、默认浅灰边框、active 统一用 `--control-active-*`。
- 适用类：`.composer-mode-toggle`、`.composer-tool-button`。
- 响应式优先按**容器宽度**判断（右侧面板打开使聊天区变窄也要触发图标模式），不只看 viewport。
- 附件预览：每个文件独立卡片（同格式也不合并）；图片显示真实缩略图（可点开大图），PDF/通用文件显示文件卡片；meta 拆成短标签可换行，不堆长串、不撑高输入区；每张卡片有移除入口（只取消本轮附件/引用，不删原文件）。区分"附件"（本轮上传/发送）与"引用文件"（已在可访问位置、本轮仅读取）。
- 已选能力提示（`/query`、公司预设 Skill、提示词）用统一轻量卡片：左能力图标、中能力名、右小号分类（`提示词` / `公司预设` / `内置命令`），使用碧落 `--control-active-*` 或 `--capability-*`，必须有取消入口。

### 6.5 列表项与侧栏项

当前项用碧落淡蓝填充；hover 用更淡同色；状态标签放次级位置，不挤压标题。

### 6.6 文件面板（右侧常驻，仅项目/客户资料；个人工作台无个人文件面板）

- 默认紧凑列表（非大图标网格）：每行小图标 + 文件名 + 大小/入库状态；文件名最大宽展示、超长末尾省略并 `title` 保留完整路径。
- 不展示底层目录（`.git`、`derived`、`manifests`、`.pending_review`、`_preprocessed`、`gbrain-ready`、`runs` 等后端产物）。`.trash` 显示为回收站入口，不可上传/重命名/拖拽。
- 预览不插入列表内部；点击文件在面板侧面打开可关闭的预览窗。
- 路径栏用 Windows 地址栏逻辑：后退/前进/上一级 + `›` 分隔层级；上传/新建为高频头部动作，刷新/回收区/录入放更多菜单。
- 颜色只复用 `--control-active-*` / `--success` / `--warning` / `--destructive` / `--muted` / `--border` / `--card`；右键菜单用轻量浮层，不引新色。
- 录入二次确认必须说明：递归处理、预计文件数、主要类型、高成本 MiMo/PDF/转写风险、当前权限。

### 6.7 确认弹窗与用户控制

关键操作（改文件、删除、覆盖、跑高风险命令、提交代码、外部请求、批处理、生成/替换正式文件）前必须确认，弹窗说明：要做什么、影响范围、风险、可选操作。主操作按钮只用于明确确认；取消/修改/查看详情为次级。可用动作词：`Approve` / `Reject` / `Edit Request` / `Run Again` / `View Details`。无法回退的操作确认前必须明示。

## 7. 反模式画廊（Gary 的痛点 → 正确做法）

| 痛点 | ❌ 错误 | ✅ 正确 |
|---|---|---|
| 莫名卡片 | 把每个页面区块都做成带边框+阴影的浮动卡片，层层嵌套 | 卡片只用于消息/计划/工具调用/文件变更/结果/Review/弹窗；大区块用分区+细边框+留白（§3.7、§6.2） |
| 颜色混乱 | 随手写 `#3b82f6`、给 active 用紫/橙、深浅主题各写一套硬编码色 | 一律 `hsl(var(--token))`；普通选中态只用碧落 `--control-active-*` |
| 字号混乱 | 同类标题忽 14 忽 22；卡片标题用 hero 字号 | 按字号阶梯取值；卡片/侧栏/工具栏禁用 hero（§3.2） |
| 排版混乱 | gap/padding 随手 7/9/13/18 乱填 | 用 4 的倍数间距阶梯；复用 `--panel-gap` 等 token（§3.3） |
| 圆角混乱 | 每个组件一个随机圆角 4/5/6/10 | 用 `--shell-radius`/`--card-radius`/`--control-radius`（§3.4） |
| 图标乱 | 相邻按钮 14 与 18 混用；图标随文字截断变形；图标乱放 | 同组统一尺寸、`flex-shrink:0`；按阶梯配字号；位置遵守布局职责（§3.5、§5.1） |
| 僵硬无动效 | 交互态直接硬切，无过渡 | hover/active/选中/展开加 `0.12–0.16s` 过渡（§4） |
| 动效过度 | 大面积动画、闪烁、高饱和多色 loading | 只用帮助理解状态的轻动效；loading 稳定中文文案（§4） |
| z-index 大战 | 到处 `z-index: 9999` | 用 `--layer-*` 命名层级（§3.6） |
| 深色主题破相 | 把浅色卡片直接塞进深色界面 | 深色用深色面板底 + 状态色边框，保留绿色语义（§3.1、§3.7） |

## 8. Agent 交付前自检清单（每次前端改动必须逐条核对）

- [ ] 改动前已读本文件，并搜索过现有类名/token/相似组件，能复用的已复用。
- [ ] 没有硬编码颜色（无 `#hex`/`rgb()` 临时色），颜色全部走 `hsl(var(--token))`。
- [ ] 普通选中/激活态只用 `--control-active-*`（碧落暮蓝），没有新引入独立亮蓝/紫/橙做普通 active。
- [ ] 字号取自字号阶梯，卡片/侧栏/工具栏/设置项标题未用 hero 字号。
- [ ] 间距用 4 的倍数阶梯（或既有 token），无随手奇值；圆角用 `--*-radius`。
- [ ] 图标尺寸取自图标阶梯，同组统一，`flex-shrink:0`，位置符合布局职责。
- [ ] z-index 用 `--layer-*`，无魔法数字。
- [ ] 新卡片确实属于"具体对象"场景，没有把普通 section 做成浮动卡片或层层嵌套。
- [ ] 交互态（hover/active/选中/展开）有 `0.12–0.16s` 过渡，不僵硬；无大面积/闪烁/装饰动效；尊重 `prefers-reduced-motion`。
- [ ] 深浅主题都验证过，深色未塞浅色卡片。
- [ ] 容器变窄时按钮自动收为图标模式，标签隐藏但图标尺寸/逻辑不变。
- [ ] 新增视觉规则的话，已先更新本文件再改 CSS。

## 9. 修改流程与验证门槛

1. 先搜索现有类名、token、相似组件，判断改动属于壳层视觉、业务语义还是 Agent 执行流体验。
2. 新增同类按钮/列表项/卡片/状态，必须复用现有 token；新增 Agent 执行组件，映射到 §6.2 既有模式。
3. 确需新视觉规则：先更新本文件，再改 CSS。
4. 逐条过 §8 自检清单。
5. 完成后在 `frontend` 运行 `bun run typecheck`；涉及打包/路由/构建配置补 `bun run build`。
6. 涉及可见 UI 的改动，用浏览器或 Electron 做一次截图 / DOM 验证（深浅主题各一次）。
