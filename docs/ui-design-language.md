# Project_R UI 设计语言与 Agent 工作台框架

本文件记录 Project_R 前端界面的可执行设计规则。新增或修改 UI 前，应先检查本文件和现有 CSS token，优先复用既有样式，不再临场自定义一套视觉状态。

本版已融合 `docs/Agent 软件 UI 设计风格框架.docx` 的核心原则：Project_R 不应只是普通聊天软件，而应是面向任务执行的 Agent 工作台。

## 设计定位

Project_R 是内部 AI 办公工具，也是面向企业业务场景的 Agent 软件。它不是营销站点，也不是单纯的问答机器人，而是一个能理解任务、拆解步骤、调用工具、处理文件、展示过程并交付结果的智能工作台。

长期 UI 原则：

- Proma shell first：参考 Proma 的布局节奏、轻量边框、侧栏与输入区质感。
- Project_R core always：业务语义、权限、文件、知识库、Skill、审计和执行链路必须服从 Project_R 后端架构。
- Codex-style execution：Agent 任务必须透明、可追踪、可确认、可复盘。

设计关键词：

- 专业、克制、高效、透明、可控。
- 工具化、低干扰、面向执行结果。
- 信息密度适中，适合长时间工作。

## 产品气质

界面应保持安静、克制、可扫描，优先服务长期办公和项目处理。

- 视觉重点不是炫酷，而是让用户清楚知道 Agent 理解了什么、正在做什么、用到了什么、改了什么、哪里需要确认、最终交付物在哪里。
- 不使用装饰性大色块、渐变球、过度动画或营销式 hero。
- 控件状态必须一眼可辨，但不能抢走聊天内容、项目资料和执行结果的视觉焦点。
- 普通 Chat 可以保留轻量对话体验；Agent 模式必须逐步演进为任务执行流，而不是普通聊天流换皮。

## 工作台布局

Project_R 主工作台采用三栏式布局，但三栏承担不同职责。

### 左侧：项目与线程导航区

左侧用于项目、工作区、会话、Agent 线程和设置入口。

- 宽度较窄，支持折叠。
- 当前项目、当前会话、当前线程必须清楚高亮。
- 展示信息保持短句，不在侧栏塞入长说明。
- 线程项应展示标题、所属项目、状态、最后更新时间和短摘要。
- 工作区列表应分为 `个人` 和 `项目` 两组：`个人` 组显示当前登录用户的 `{用户名}的工作台`，`项目` 组显示公司项目工作区。
- `{用户名}的工作台` 使用个人图标，项目工作区使用项目或文件夹图标；两者不能只靠文字区分。

### 中间：Chat / Agent 执行区

中间是核心区域。普通 Chat 展示对话；Agent 模式展示任务执行流。

Agent 任务建议按以下结构组织：

- 用户目标。
- Agent 理解。
- 执行计划。
- 执行过程。
- 工具调用摘要。
- 文件变更摘要。
- 需要确认的问题。
- 最终结果。
- 后续建议。

默认只展示高层摘要和当前步骤，日志、完整命令输出、完整 diff 和历史上下文通过展开查看。

### 右侧：上下文与结果预览区

右侧用于当前任务相关内容，不做装饰面板。

- 文件预览、代码 diff、Markdown 引用片段、运行结果、测试结果、任务 checklist、错误详情和输出文档预览都应优先出现在右侧。
- 右侧面板必须绑定当前会话或当前任务，切换会话时自动关闭或刷新上下文，避免误读旧来源。
- 不要把预览插入文件列表内部；列表和预览职责分开。

## Agent 任务流程

输入框不只是聊天输入，而是任务目标入口。用户提交 Agent 任务后，界面应鼓励以下流程：

1. 用户选择项目或上下文。
2. 用户输入任务目标。
3. Agent 显示任务理解。
4. Agent 生成执行计划。
5. 用户确认或修改计划。
6. Agent 执行任务。
7. 系统展示过程和变更。
8. Agent 生成结果。
9. 用户 review。
10. 用户接受、修改、回退或重新运行。

当前阶段如后端能力尚未闭环，可先展示对应 UI 占位，但必须标明真实状态，不能把未接入能力伪装成已执行。

## 信息层级

Agent 软件容易信息过载，必须按层级展示。

第一层：用户最关心的信息。

- 当前任务是否完成。
- Agent 正在做什么。
- 是否需要用户确认。
- 最终结果是什么。

第二层：任务过程。

- 执行计划。
- 当前步骤。
- 工具调用摘要。
- 文件修改摘要。

第三层：技术细节。

- 命令行输出。
- 代码 diff。
- 错误日志。
- 测试详情。

第四层：完整历史。

- 所有工具调用。
- 所有中间输出。
- 完整上下文。
- 版本记录。

默认展示第一层和第二层。第三层和第四层只能通过展开、详情页或 Review 页查看。

## 状态模型

Agent 线程、执行步骤、工具调用和文件变更应使用稳定状态，不要每个组件自造状态词。

### Thread 状态

| 状态 | 含义 |
|---|---|
| `Draft` | 用户刚输入，尚未执行 |
| `Planning` | Agent 正在制定计划 |
| `Running` | Agent 正在执行 |
| `Waiting Approval` | 等待用户确认 |
| `Need Review` | 结果需要用户检查 |
| `Failed` | 执行失败 |
| `Completed` | 已完成 |
| `Archived` | 已归档 |

### Step / Tool 状态

| 状态 | 显示方式 |
|---|---|
| 等待中 | 灰色圆点或 muted 文案 |
| 执行中 | 轻量 loading，不闪烁 |
| 已完成 | 绿色勾号或 success tag |
| 需要确认 | 黄色提示 |
| 出错 | 红色警告 |

状态标签应小尺寸、浅色背景、低饱和，不要抢正文视觉焦点。

## 核心组件规则

### 任务理解卡片

Agent 执行前应先展示简短理解，避免用户不知道系统是否理解正确。

内容包括：

- Agent 对用户目标的复述。
- 本次任务会影响的范围。
- 不确定点或需要用户确认的边界。

### 计划卡片

计划卡片用于让用户判断 Agent 的执行方向。

内容包括：

- 任务目标。
- 执行步骤。
- 预计涉及文件或资料。
- 可能风险。
- 是否需要用户确认。

计划卡片不要写成长文档，步骤应短、可扫读。

### 执行步骤卡片

每个步骤是一张小卡片或时间线项。

- 展示步骤名称、状态、简短说明和展开详情入口。
- 执行中步骤在视觉上略突出。
- 已完成步骤收敛为摘要，避免占用过多空间。

### 工具调用卡片

工具调用不应直接倾倒大段日志，而应用工具卡片表达。

内容包括：

- 工具名称。
- 输入对象，例如文件、命令、URL、知识源。
- 输出摘要。
- 状态。
- 展开日志入口。

工具卡片默认折叠，失败时可自动展开错误摘要。

### 文件变更卡片

Agent 修改文件时必须展示可 review 的文件变更。

内容包括：

- 文件名和路径。
- 修改类型。
- 修改摘要。
- 查看 diff。
- 接受、回退或重新生成入口。

文件变更不能只出现在最终回答里；需要进入可复盘的任务记录。

### 结果卡片

任务完成后展示结果卡片。

内容包括：

- 完成状态。
- 主要修改或产物。
- 输出文件。
- 测试或验证结果。
- 建议下一步。

如果未运行测试或无法验证，必须明确显示“未验证”而不是隐藏。

### 输入框附件与引用预览

输入框附件体验应接近成熟聊天软件的操作逻辑：用户把文件拖入输入区后，必须立即看到可识别、可预览、可移除的附件预览，而不是只显示一行受限文件名。

- 图片附件显示真实缩略图，点击缩略图可打开大图预览，卡片右上角提供取消附件按钮。
- PDF 和不可生成图片缩略图的文件显示文件卡片，包含文件图标、文件名和类型；PDF 应支持点击预览查看。
- 输入框预览区必须区分“附件”和“引用文件”。附件表示本轮要上传或发送的文件；引用文件表示文件已在项目资料、客户资料或其他可访问位置，本轮只让 AI 读取。
- 每个文件都必须显示为独立卡片；即使文件格式相同，也不能合并成一个汇总 chip。
- 预览能力第一版只承诺简单图片和 PDF。Office 文件、压缩包、音视频和其他通用文件只显示文件卡片、文件类型和基础信息，不承诺在输入框中预览内容。
- 当图片缩略图与文件卡片混排时，图片缩略图高度应与文件卡片高度协调，不应因为混排造成输入框高度或卡片比例失控。
- 附件预览区位于输入框内容上方或输入框内部上沿，保持横向排列和可滚动，不遮挡输入文字、发送按钮、模型选择或麦克风等操作。
- 每个附件卡片都必须有明确移除入口；移除动作只取消本次输入的附件或引用，不删除项目资料、客户资料或本机原文件。
- 文件右键菜单中的“引用文件”会把文件加入输入框顶部引用预览；引用预览与附件预览共享卡片语言，但应标识来源，例如本机选择文件、项目资料、客户资料。
- 所有文件默认不被 AI 读取；只有出现在本次输入的附件预览或引用预览中，或通过显式知识库查询/分析动作提交后，才可进入模型上下文。
- 对话气泡导出和文件下载统一保存到本地。Project_R 不再提供“导出到工作台”或“下载到工作台”的个人文件流。

发送后，用户消息气泡中仍需保留本轮附件和引用文件的紧凑卡片摘要，方便回看当时给 AI 的上下文。

- 用户消息气泡可先展示附件/引用卡片，再展示用户文本；当用户只发送文件和短文本时，文件卡片应成为该消息的主要视觉对象。
- 图片文件在消息气泡中保留缩略图，点击可查看大图。
- PDF 和通用文件在消息气泡中保留文件卡片，显示文件名、类型和来源；PDF 可点击预览。
- 消息气泡中的文件卡片不再显示输入态的取消按钮；个人工作台内只提供预览、复制内容和下载到本地等动作，不提供保存到项目/客户资料。项目/客户工作区内的文件卡片动作按当前工作区治理规则裁剪。
- AI 消息如果实际读取了附件或引用文件，应在回答正文或来源区显示紧凑来源 pill，点击后定位到对应用户消息文件卡片或打开文件预览。
- AI 回答的文件来源采用两层显示：正文内可插入紧凑来源 pill 做就近说明，消息底部保留统一“引用来源”列表集中展示本次实际读取的文件。
- 正文内来源 pill 应短小、低干扰，只显示截断文件名或来源别名；底部来源列表可显示更完整的文件名、来源类型、所属位置和预览入口。
- 文件卡片摘要必须保持克制，避免把聊天流撑成文件管理页；长文件名应截断并提供 hover/title 或详情入口。

右侧常驻文件面板只用于项目资料、客户资料等共享或受限资料空间；个人工作台不再提供个人文件面板。需要长期保存、共享、权限治理、审计、入库、复盘或被项目/客户知识库引用的业务文件，只能进入项目/客户文件面板。

- 项目资料基础动作包括新建文件夹、重命名、删除、移动、复制、下载到本地、预览、引用文件、上传文件。
- 项目资料专属动作包括一键录入项目知识库、查看项目 source 状态、项目权限和审计相关操作。
- 不要在个人工作台显示右侧个人文件面板、常用文件、对话文件、个人回收站或个人 30 天文件清理状态。
- 个人工作台可以显示轻量业务 Skill / Agent 生成任务的运行卡片和结果卡片，例如起草、改写、模板填写、生成可下载文件；这些卡片不得带出项目/客户文件治理、GBrain 入库、source 管理、知识审核、客户画像或图谱实体合并控件。
- 个人工作台中的轻量业务 Skill / Agent 输出默认只作为本轮结果展示；`下载到本地` 与 `另存为本地文件` 是同一个保存到本机动作，不应拆成两个功能。
- 不要把导出、下载、会话附件、本机选择文件或 AI 生成文件保存到个人工作台、项目资料或客户资料。项目相关工作应在项目工作区完成，客户相关工作应在客户工作区完成。
- 项目/客户工作区里的业务 Skill / Agent 输出也先作为本轮结果卡片展示；只有用户确认保存后，才写入当前工作区文件面板，默认进入当前工作区 `99-未归档文件`。
- 工作区 Skill 输出的“保存到当前工作区”动作必须明确显示目标工作区和保存目录；保存成功后才展示文件卡片、审计状态和后续入库候选状态。保存动作不得自动触发 GBrain 入库。
- 项目工作区右侧文件面板打开时默认显示项目资料根目录，不按当前会话自动定位。
- 项目/客户文件面板只展示用户源文件、回收站状态和可操作文件状态；不得展示 `_preprocessed/`、`gbrain-ready/`、`runs/`、`manifests/`、GBrain sidecar 或其他后端过程产物。
- 文件面板顶部“录入”动作默认处理当前打开路径下的文件和子文件夹；执行前必须弹出二次确认，说明这是递归处理、预计文件数量、主要文件类型、高成本 MiMo/PDF/转写风险和当前用户权限。
- 文件右键菜单应提供“录入此文件”“忽略录入 / 取消忽略”“重新录入此文件”等动作；普通用户能看到的动作必须按当前工作区权限裁剪。
- 项目工作区中，系统管理员和项目管理员可以触发文件夹递归录入和单文件录入；普通成员只能对自己上传的单个文件触发录入，不能触发递归文件夹录入。
- 客户工作区中，只有系统管理员和客户工作区管理员可以触发客户情报录入；普通成员可以上传和查看有权限的客户资料，但不能启动客户情报提炼或 GBrain 写操作。
- 录入状态应使用稳定词：`queued`、`preprocessing`、`gbrain_ready`、`sync_pending`、`synced`、`failed`、`pending_capability`、`ignored`、`source_changed`、`needs_repreprocess`。状态文案要解释“源文件已删除不等于知识已删除”，避免用户误以为删除源文件会清空 GBrain。

### Review 页 / Review 面板

Review 用于检查 Agent 输出，不是普通详情页。

适合展示：

- 修改前后对比。
- 代码 diff。
- 文档修改痕迹。
- 自动总结。
- 风险提示。
- 接受、退回、重跑按钮。

## 用户控制与信任

Agent 可以执行任务，但关键操作必须保留用户控制权。

需要确认的操作：

- 修改文件。
- 删除内容。
- 覆盖原文件。
- 运行高风险命令。
- 提交代码。
- 发送外部请求。
- 批量处理数据。
- 生成最终版本或替换正式文件。

确认操作可使用：

- `Approve`
- `Reject`
- `Edit Request`
- `Run Again`
- `View Details`

所有修改应支持查看记录、回到上一版本、拒绝本次修改或重新生成结果。无法回退的操作必须在确认前明确提示。

权限等级建议：

| 权限等级 | Agent 能做什么 |
|---|---|
| `Read Only` | 只能读取和分析 |
| `Suggest` | 可以提出修改建议 |
| `Edit with Approval` | 修改前需要确认 |
| `Auto Execute` | 可自动执行低风险操作 |

## 页面模块

### 首页 Dashboard

首页是工作入口，不是营销欢迎页。

应优先展示：

- 最近项目。
- 最近任务。
- 未完成线程。
- Agent 推荐操作。
- 快捷新建任务。
- 最近输出文件。

### 项目页 Project Workspace

项目页是核心工作台。

- 左侧：项目文件和任务线程。
- 中间：Chat / Agent 对话和执行过程。
- 右侧：文件预览、来源、结果和 Review。
- 项目工作区 `/query` 查询范围是 `company-wiki + 当前项目 source`；普通 Chat 不自动查 GBrain。
- 后续可增加“项目洞察”入口，用于项目摘要、时间线、风险、决策、联系人、证据和复盘视图；该入口只能作为 Project_R 对 GBrain 原生能力的受控转发和展示，不得另建一套知识库。

### 客户情报页 Customer Intelligence

客户工作区需要一个面向销售和客户管理场景的“客户情报”入口，而不是把客户工作区简单做成项目文件页换名。

- 建议标签页：`画像概览`、`图谱`、`时间线`、`实体处理`、`GBrain 状态`。
- `画像概览`、`图谱`、`时间线` 和只读来源预览可向普通授权成员开放；`实体处理`、Entity Enrichment、实体合并、source sync、graph merge 等写操作只向系统管理员和客户工作区管理员开放。
- 面板操作本质是 Project_R 对 GBrain 原生 schema、enrich skill、MCP operation、CLI/API、graph/timeline/backlinks 能力的转发和可视化，不得在 Project_R 内重做一套 People Graph、Company Graph 或 Project Graph 知识内核。
- 客户工作区 `/query` 只查询受限客户情报 GBrain 数据，不叠加 `company-wiki` 或项目 source；UI 文案应使用“客户情报”“客户画像”“客户图谱”，不要把 `customer-reference` 当成产品概念展示。
- 公司全局知识库不提供普通用户“知识地图 / 公司知识洞察”入口；普通用户使用 `/query` 或知识库 Skill，管理员使用 GBrain 状态、回归、维护和审核入口。

### 执行页 Agent Run

执行页用于展示 Agent 正在工作的过程。

核心模块：

- 当前目标。
- 执行计划。
- 当前步骤。
- 工具调用。
- 文件变更。
- 用户确认。
- 最终结果。

### 设置页

设置页用于控制 Agent 行为边界。

适合配置：

- Agent 名称和默认语气。
- 默认输出格式。
- 是否自动执行。
- 是否需要确认。
- 文件访问范围。
- 工具权限。
- 主题颜色。
- 快捷命令。

设置页不要混入普通任务结果或营销说明。

## 颜色规则

### 主色与主题

Project_R 支持浅色与深色主题。Agent 工作台长期应保证深色模式可作为长时间工作的一等体验，但现阶段不得破坏既有浅色 Proma shell。

颜色使用原则：

- 普通信息使用灰白 / 灰黑体系。
- 重点操作使用主强调色。
- 成功、失败、等待、运行中使用固定状态色。
- 避免大面积高饱和颜色。
- 保持整体冷静、专业、技术感。

### 默认选中态

Project_R 的普通选中态、激活态和当前项状态统一使用柔和浅绿色。

CSS token 位于 `frontend/src/renderer/styles.css`：

```css
--control-active-bg
--control-active-bg-hover
--control-active-border
--control-active-foreground
--control-active-shadow
```

适用范围：

- 当前工作区、当前会话、当前标签或列表项。
- 输入框功能按钮的开启态，例如深度思考、联网搜索、提示词、技能。
- Skill 候选项 hover / active。
- 轻量功能入口 hover，例如新建工作区图标。

禁止事项：

- 不要为普通 active 状态临时引入蓝色、紫色、深黑、橙色或其他新色。
- 不要让同一工具栏内的 active 状态出现多套颜色逻辑。
- 深色主题也必须保留“绿色语义”，使用暗色变量，不改成纯白或纯黑 active。

### 语义色

只有语义明确时才使用其他颜色：

- 红色：删除、失败、危险、强警告。
- 黄色 / 橙色：等待、注意、需要确认。
- 绿色：完成、成功、测试通过。
- 蓝色：外链、信息提示、已选能力提示或明确网络/链接语义，但不能替代普通选中态。
- 深色实心：主按钮或发送按钮，不用于普通 toggle active。

## 字体与排版

字体分工：

| 内容类型 | 字体建议 |
|---|---|
| 普通说明文字 | Inter / SF Pro / Segoe UI / Arial |
| 代码内容 | JetBrains Mono / Fira Code / Consolas |
| 文件路径 | 等宽字体 |
| 命令行输出 | 等宽字体 |
| 状态标签 | 无衬线字体 |

排版规则：

- 标题清晰，但不要过大。
- 正文保持 `14px` 到 `16px`。
- 代码、路径和日志保持 `13px` 到 `14px`。
- 卡片之间留出足够间距。
- 重要状态使用标签或图标辅助识别。
- 长内容默认折叠，用户需要时展开。
- 不要用 hero 级字号处理卡片、侧栏、工具栏和设置项标题。

## 控件规则

### 输入框底部功能按钮

输入框底部的功能按钮是一组同级工具，不按功能来源拆成不同视觉系统。

- 高度：`32px`
- 圆角：`8px`
- 默认边框：浅灰 `#e4e4e7`
- 默认背景：白色或暗色模式下半透明深灰
- active：统一使用 `--control-active-*`
- 文案按钮保留图标 + 短文字；当输入区或工具栏容器变窄时，必须自动隐藏文字，只保留图标，尺寸和 active 逻辑不变。
- 响应式判断应优先使用容器宽度，而不是只依赖全局 viewport 宽度；右侧面板打开导致聊天区变窄时，也必须触发图标模式。
- 同一行里的模型选择按钮也要遵守压缩规则：空间不足时只保留模型/设置图标，完整模型名通过 tooltip 或 aria-label 保留。
- 隐藏文字时只能隐藏显式 label class，例如 `.composer-button-label` / `.composer-model-label`；不要写 `button span { display: none; }`，因为图标组件也可能渲染为 `span`。

当前适用类：

- `.composer-mode-toggle`
- `.composer-tool-button`

### 输入框会话附件

会话附件是“本次对话临时上下文”，不等同于个人文件面板、私人空间同步，也不等同于项目资料入库。

- 纸夹按钮、剪贴板粘贴和拖拽到对话区默认创建 `session_upload` 临时附件，可直接随本条消息发送。
- 从本机选择文件时，跨越本机边界前必须显示来源和授权说明，说明发送形式、目标范围和保留策略。
- 普通会话附件卡片使用既有柔和浅绿色 active token；本机待确认附件可使用 `--warning` 语义 token，不能新增一套附件专属配色。
- 图片附件应显示小缩略图，文本/PDF/通用文件显示稳定类型图标；文件名和元信息必须可截断，不能撑开输入区。
- 拖拽遮罩只表达“释放以添加到当前对话”，不得把普通拖拽附件描述为个人文件入库。

### 输入框已选能力提示

当用户为本次发送显式选择某个能力，例如企业 Skill 或系统级知识库查询入口 `/query`，输入框内应展示一条独立的“已选能力提示卡片”。

- 该卡片用于表达“本次消息已绑定能力 / 上下文”，不是普通按钮 active 状态。
- 可参考 Codex 输入框选中 Skill 后的轻量信息行：小字号、蓝色文字、能力图标 + 能力名称。
- 提示词、企业 Skill 和 `/query` 使用同一套卡片结构：左侧能力图标，中间能力名称，右侧小号分类文字。分类文字统一使用 `提示词`、`Skills`、`内置命令`。
- 选择 `/query` 这类内置命令后，不应继续把 `/query` 命令文本留在输入区；输入区展示用户真实问题，发送时由前端自动附加路由指令。
- 蓝色只允许用于这类已选能力提示，不用于右侧 Skills 列表项、普通工作区项或工具按钮 active 状态。
- 右侧 Skills 面板列表仍使用默认浅绿色 hover / active 规则。
- 已选能力提示必须提供取消入口，且视觉上应与对话正文和普通输入占位文字清楚区分。

### 列表项与侧栏项

工作区、Skill 行、提示词行、线程行等列表项应保持轻量：

- 当前项用浅绿色填充，不加重边框。
- hover 可以使用同一浅绿色的更淡版本。
- 图标底色可使用浅绿色，但不要使用高饱和渐变，除非是明确品牌或文件类型图标。
- 状态标签放在次级位置，不能挤压标题。

### 项目文件路径栏

项目文件面板的路径导航采用 Windows 文件夹地址栏逻辑，导航操作和路径必须在同一条栏内完成。

- 导航按钮顺序：后退、前进、上一级。
- 路径区域使用独立地址栏容器，根目录和层级路径用 `›` 分隔。
- 图标按钮保持正方形，禁用态只降低透明度，不改变布局尺寸。
- 上传、新建是头部高频动作；刷新、回收区、知识库录入等低频动作放入更多菜单，不和路径导航混排。
- 不向普通用户暴露“移动到某路径”这类内部路径输入；文件移动应使用剪切 / 粘贴或单项拖拽等熟悉的文件夹交互。

### 项目文件列表与预览

右侧项目文件常驻面板空间有限，默认使用紧凑列表，不使用大图标网格。

- 文件夹和文件每行一条：左侧小图标，中间文件名，右侧大小和入库状态。
- 文件名必须允许最大宽度展示，超长时末尾省略，并通过 `title` 保留完整路径。
- 不展示底层系统目录，例如 `.git`、`derived`、`manifests`、`.pending_review`。
- `.trash` 必须在普通文件目录中显示为回收站入口；点击后进入回收站视图，可访问当前工作区的用户都能还原文件。
- `.trash` 入口不能呈现为普通可编辑目录：不允许上传、拖拽、重命名、复制、移动或删除。
- 预览不得插入列表内部；点击文件后在文件面板侧面打开可关闭的预览/详细信息窗。
- 颜色只能复用现有 token：`--control-active-*`、`--success`、`--warning`、`--destructive`、`--muted`、`--border`、`--card`。
- 右键菜单采用轻量浮层，复用当前文件操作，不引入新的色彩体系。
- 右键菜单按 Windows 文件夹心智组织高频项：预览、打开、剪切、复制、粘贴、重命名、删除、刷新、新建文件夹、详细信息。
- 当没有待粘贴对象、目标目录不可写、权限不足或文件正在处理时，对应菜单项禁用而不是隐藏，避免用户误以为功能消失。
- 第一版不做多选文件操作；拖拽仅移动当前拖起的单个文件或文件夹。
- 拖拽目标可以是文件夹行或路径栏层级，hover 使用既有柔和浅绿色选中态，不新增拖拽专用配色。

### 确认弹窗

当 Agent 准备进行关键操作时，确认弹窗必须说明：

- Agent 准备做什么。
- 影响范围。
- 风险提示。
- 用户可选操作。

主操作按钮只能用于明确确认。取消、修改指令、查看详情应作为次级操作。

## 边框、阴影与卡片

- 业务工具界面优先用细边框和留白分隔。
- 浅色主题边框使用低对比灰色。
- 深色主题边框透明度要低，避免线框感过重。
- 阴影只用于浮层、弹窗、输入框容器和 active 工具按钮的轻微反馈。
- 不要把页面 section 设计成层层嵌套卡片。
- 卡片用于任务计划、工具调用、文件变更、结果、Review 项等具体对象。

## 动效规则

动效只用于帮助用户理解状态变化。

允许：

- Agent 执行中的轻微 loading。
- 步骤完成时的轻微状态变化。
- 卡片展开的平滑过渡。
- 新结果生成时轻微淡入。
- 错误提示轻微高亮。

避免：

- 大面积动画。
- 强烈闪烁。
- 复杂转场。
- 装饰性动效过多。

## 修改流程

1. 先搜索现有类名、token 和相似组件。
2. 判断改动属于 Proma shell 视觉、Project_R 业务语义，还是 Agent 执行流体验。
3. 若只是新增同类按钮、列表项、卡片或状态，必须复用现有 token。
4. 若新增 Agent 执行组件，优先映射到本文件的任务理解、计划、步骤、工具调用、文件变更、结果或 Review 模式。
5. 若确实需要新增视觉规则，先更新本文件，再改 CSS。
6. 完成后至少运行 `frontend` 的 `bun run typecheck` 和 `bun run build`。
7. 涉及可见 UI 的改动，需要用浏览器或 Electron 做一次截图/DOM 验证。
