### 一、项目背景与目标

公司内部拥有大量专业领域的标准、规范与业务数据（目前约 200 个文件，预计年内增至 500 个）。为提升全员（约 40 人）的日常查阅效率与文件制作质量，计划研发一套 C/S 架构的 AI 智能办公辅助系统。

本系统的核心目标是提供一个极简的桌面端统一入口，实现"四位一体"的办公闭环：

1. **通用 AI 问答** — 自由对话与语义解释
2. **企业知识库精准检索（RAG）** — 基于公司内部文件的准确解答，避免幻觉
3. **办公文件全自动生成** — 当前通过显式入口 / 业务 Skill 触发本地脚本，长期恢复自然语言意图触发，生成 Word/PPT/Excel 成品
4. **业务工作流 Agent / Skill** — 按公司业务流程节点拆分预设的 Skill 与 Agent 工具，实现工作流的全面数字化增效

第 4 项是本系统区别于通用 AI 助手的核心能力，也是相比开源参考项目 Proma 的主要差异化扩展点。

### 三、系统架构与核心技术栈

本系统采用"本地物理算力 + 云端安全接口"的混合架构：

- **网络拓扑**：Electron 前端运行于员工内网 PC，通过局域网与 Mac mini 后端通信；Mac mini 后端通过受控出网通道调用云端 LLM Provider（默认 Claude，预留 OpenAI / ChatGPT、DeepSeek、MiMo 等厂商）。员工客户端不直接接触任何 API Key 或云端接口。
- **部署模式**：内网局域网封闭运行（仅后端服务器具备出网权限）。
- **架构原则**：**前端是壳子，所有业务逻辑在后端**。Electron 不复用 Proma 的 IPC 主进程编排架构，前端代码第一阶段通过 HTTP 调用 Mac mini 后端接口；SSE 流式能力后续作为独立端点追加，避免破坏现有 JSON 消息底座。

#### 前端（客户端）

| 项 | 选型 |
|---|---|
| 桌面框架 | Electron |
| 前端框架 | React + Vite + TypeScript |
| 状态管理 | **Jotai**（参考 Proma；按功能拆分到 `atoms/` 目录） |
| UI 组件 | 初步雏形阶段以 Proma UI shell 为强参考基座；已获授权前提下，允许局部迁移或强复刻 Proma 的布局、组件观感与交互节奏，但业务逻辑、数据流与安全边界必须按 Project_R 自实现 |
| 默认窗口尺寸 | 1400×900；最小 800×600；记住上次窗口位置；多显示器自适应 |
| 窗口控制 | 取消系统主菜单栏；主工作台将最小化 / 最大化 / 关闭按钮融入标签栏右侧，登录、设置、欢迎等无标签页页面提供极简拖拽兜底层 |
| 服务器连接 | 永远从配置读取后端 IP / 端口，**请求层不写死 localhost**；开发期默认值来自 `frontend/.env.development` 的 `VITE_DEFAULT_API_BASE_URL`，部署期由安装包或本地配置覆盖 |

#### 后端（无头服务器）

| 项 | 选型 |

| 服务框架 | Python FastAPI + Uvicorn |

| 向量检索 | **ChromaDB**（自带增删改查 API，便于增量索引管理） |
| 文件渲染 | `python-docx`、`python-pptx`、`openpyxl` |
| 文件解析 | `pypdf2`、`python-docx`、`officeparser` 等（支持 md/docx/pptx/xlsx/PDF） |
| 数据库 | SQLite（用户、会话、审计日志、知识审核队列等） |
| Skill / Agent | 沿用 Proma 的 Skill 格式，与 Claude Agent SDK 兼容 |

### 四、核心功能需求

#### 1. 意图分流与对话系统 (Intent Recognition & Chat)

- **需求描述**：系统具备单一聊天流界面。长期目标是在后端实现自动意图分流；当前阶段临时采用全部显式路由，避免半成品 Skill / 文件生成能力误触发。
- **业务逻辑**：
  - 当前实现：`core/intent.py` 默认返回 `chat`；知识库问答必须使用 `/query ...`；业务 Skill 必须由用户在前端 Skill 面板手动选择；文件生成需后续明确入口或 Skill 承接。
  - 长期实现：接收到用户输入后，利用大模型的 Function Calling 或前置判定逻辑，识别当前任务属性（普通问答 / RAG 检索 / 文件生成 / 业务 Skill 触发）。
  - 若为"普通问答"，直接调用通用大模型能力。当前稳定接口返回 JSON 完整回复，前端用打字机效果模拟逐字出现；真正流式（Streaming）后续作为独立 SSE 能力补充。
  - 多轮对话上下文存储于后端，按用户身份（User ID）严格隔离，互不干扰。

##### 1.1 会话级提示词系统

提示词调用语义为**当前会话的 system prompt**，不是把提示词文本插入输入框。每个会话可独立选择一个提示词，新会话默认使用 `Project_R 内置提示词`。

提示词按来源拆为三类：

| 来源 | 存储位置 | 用户权限 | 说明 |
|---|---|---|---|
| Project_R 内置提示词 | 随系统发布，代码或后端配置内置 | 普通用户只读查看和调用 | 定义 Project_R 默认人格、回答纪律、公司 AI 助手边界 |
| 公司预设提示词 | 后端数据库保存 | 管理员维护；普通用户查看和调用 | 用于公司统一工作流，如项目复盘、会议纪要、资料审查、邮件润色 |
| 用户自定义提示词 | Electron 主进程写入 `app.getPath("userData")` 下的 JSON 文件 | 仅当前本机用户可新增、编辑、删除和调用 | 不进 Git，不上传后端，不跨设备同步 |

当前实现已完成聊天工作台内提示词调用面板与设置页提示词管理入口：提示词选中后只显示会话级 chip，不改写输入框正文；公司预设提示词由后端保存，用户自定义提示词由 Electron 主进程保存到本机 `userData` JSON。

#### 2. 企业知识库精准检索 (RAG Mode)

> **知识源设计（2026-05-19 决策）**：知识库文件直接存放在 `backend/knowledge_base/` 目录下，首期为 `rules/` 和 `training/` 两个子目录（源自公司 Obsidian Wiki 的结构化 .md 文件）。管理员通过管理接口手动触发"刷新知识库"，RAG 引擎扫描本地目录、按 `##` 标题分块 + 提取 YAML 前置元数据（tags / authority_level / content_kind）、sentence-transformers 本地模型向量化后存入 ChromaDB。不做外部同步。

实现状态（2026-05-24，2026-05-27 路由策略更新）：RAG 10A 已完成本地可验证底座，测试覆盖 `.md` 扫描、YAML frontmatter、标题分块、hash 跳过、删除同步和来源片段返回。10B 已完成代码级最小接入：管理员 refresh/status 权限、固定意图枚举、`/query ...` 显式知识库问答时检索来源并注入当前 LLM system prompt，assistant 消息持久化 RAG 来源摘要，前端消息区展示来源。当前临时采用全部显式路由，普通自然语言聊天不自动推断为 `rag_query`；后续待 Skills / Agent 执行链路完整后再恢复“Chat 发现意图”。当前本机已确认安装 `chromadb`、`sentence-transformers`、`pyyaml`，本地模型缓存和向量库目录存在；项目资料侧已支持当前项目文本类资料轻量召回。真实 Office/PDF 项目资料解析、分块、向量化和来源文件预览仍为下一阶段。

- **需求描述**：对公司内部文件进行深度检索，避免 AI 产生幻觉。
- **支持格式**：Markdown（首选）、Word（.docx）、PowerPoint（.pptx）、Excel（.xlsx）、PDF。
- **格式策略**：以 **Markdown 为主格式**，与现有 Obsidian 知识库工作流对齐；其他格式作为"原始素材"支持导入，但不对排版保真度做承诺。
- **业务逻辑**：
  - 后台对指定目录下的文件完成自动化切片与向量化存储（ChromaDB）。
  - 命中知识库问答时，优先检索本地库，将高相关性文本作为上下文（Context）投喂给 AI 进行总结提炼。

##### 2.1 知识库分类（多维度）

参考现有的 LLM Wiki 架构（`Gary-WIKI/wiki/` 下分 `rules/`、`standards/`、`sources/`、`tags/`、`training/` 等子目录）：

- **全局知识库**：公司通用规则、规范、模板、培训资料
- **项目子库**：每个项目可建立独立子库（项目专属图纸、合同、热工报告、签认资料等）
- **多维度标签**：单个文档可同时归属多个标签（项目 / 类型 / 年份 / 业务领域）
- **检索方式**：支持"全局检索"、"项目内检索"、"跨项目对比检索"三种模式

##### 2.2 知识库更新策略（本地目录 + 增量索引）

- 知识库文件直接存放在 `backend/knowledge_base/` 目录下，子目录列表（`RAG_SOURCE_DIRS`）由 `.env` 配置（默认 `rules,training`）。
- 管理员在后台点击"刷新知识库"，系统扫描本地目录中的 `.md` 文件，分块、向量化并存入 ChromaDB。
- 系统为每个已索引文件记录文件哈希值与最后修改时间戳，仅对新增或变更的文件重新向量化，未变更文件跳过（增量更新）。
- 从目录中删除的文件，下次刷新时自动从向量库中移除对应条目。

##### 2.3 会话临时附件（区别于知识库）

- 普通用户在对话中可上传临时文件（合同、图纸、参考资料等），仅当前会话可见，**不进入持久化向量库**。
- 用于一次性分析任务（如"帮我审一下这份合同"），不污染全员知识库。
- 参考 Proma 的附件机制实现。

##### 2.4 工作区项目资料目录（项目子库来源之一）

工作区在 Project_R 中不是单纯 UI 分组，而是按公司项目组织的**项目资料容器**：

- 用户通常按项目名创建工作区；后端为每个工作区创建对应的专用目录。
- 工作区目录用于保存该项目的参考资料，例如图纸、合同、报价、会议纪要、变更、生产发货资料、现场与客诉资料等。
- Agent 模式默认显示当前工作区的文件目录/上下文面板，让用户确认参考文件是否正确、是否遗漏；Chat 模式只提供轻量入口，不在侧栏常驻完整文件树。
- 第一版仅允许操作后端创建的工作区专用目录，不允许用户绑定任意本机目录，避免误删本机文件、泄露非项目资料和跨平台路径失控。
- 当前已完成默认项目资料目录、文件树接口、Agent 目录面板、Chat 轻量入口、多文件上传、100MB 单文件限制、删除进入工作区回收区、恢复/永久删除、成员/管理员删除权限、文件操作审计与项目资料索引 pending/indexed 状态。
- 项目对话已按作用域接入当前项目、未删除、已索引的文本类资料轻量召回；其他项目和无项目对话不能访问该项目资料。
- Office/PDF/Excel 等办公格式当前可作为项目文件上传入库，但要进入问答上下文仍需后续解析、分块、向量化和来源定位。
- 工作区创建时自动生成固定默认目录，后续再按项目类型模板化。建议初始目录：`01-合同与报价/`、`02-图纸与技术资料/`、`03-会议纪要/`、`04-变更与签证/`、`05-生产与发货/`、`06-现场与客诉/`、`99-临时资料/`。

工作区文件管理采用以下权限策略：

- 工作区成员可上传文件。
- 普通成员只能删除自己上传的文件。
- 工作区管理员可删除所有文件。
- 删除优先软删除或移入工作区回收区，不直接物理删除。
- 上传、删除、恢复等操作必须写入审计日志。
- 支持有限层级目录：显示目录树、新建文件夹、上传到指定文件夹、删除文件；删除文件夹仅允许空文件夹；不做拖拽移动和递归删除。

##### 2.5 知识库维护权限

- **刷新索引**：仅管理员可触发"刷新知识库"（扫描本地目录 + 重建向量索引）。
- **维护文件**：知识库 `.md` 文件由管理员直接维护（增删改）在 `backend/knowledge_base/` 目录下，修改后需触发刷新使变更生效。
- **蒸馏入库**：AI 在对话中提炼出的规则知识候选，进入"知识审核区"（SQLite `knowledge_review` 表），由管理员审核后写入知识库目录并触发刷新索引。（参见第 6 节）。

#### 3. 办公文件全自动生成 (Automated Document Export)

- **需求描述**：无需员工掌握复杂的排版软件，通过对话一键输出成品文件。
- **输出形式**：根据任务性质支持两种输出方式，由 AI 自动判断或用户指定：
  - **聊天界面文本响应** — 适合简短意见、审核反馈、解释说明
  - **生成成品文件**（.docx / .pptx / .xlsx）— 适合长篇报告、标准化业务文档
- **业务逻辑**：
  - 当前显式路由阶段：普通聊天中的"生成文件"自然语言不自动触发文件生成；后续由明确文件生成入口或业务 Skill 承接。
  - 长期恢复自动意图识别后：识别到"生成文件"指令，拦截大模型的纯文本/Markdown 响应。
  - 触发后端 Python 排版脚本，将文本动态填入公司预设模板（存放于 `templates/` 目录）。
  - 文件落盘于服务器本地后，向 Electron 前端发送下载指令与链接，前端弹出卡片供用户保存至桌面。
- **临时文件清理**：生成的文件保留 **48 小时**，后台定时任务自动清理过期文件。
- **错误处理**：
  - 模板缺失：返回明确错误提示"未找到对应模板，请联系管理员"，不静默失败。
  - 渲染异常：捕获 `python-docx`/`python-pptx` 异常，记录错误日志，向用户返回"文件生成失败，已记录错误，请重试或联系管理员"。
  - 磁盘空间不足：生成前预检可用空间，低于阈值时拒绝任务并告警。
  - 所有失败均不影响当前对话会话的继续使用。

#### 4. 用户身份验证与权限控制 (Auth & RBAC)

- **需求描述**：系统区分普通员工与管理员，保护后端接口与知识库管理功能。
- **角色定义**：
  - **普通用户**：登录后可使用聊天、RAG 查询、文件生成、调用业务 Skill 等核心功能。
  - **管理员**：在普通用户权限基础上，额外拥有用户管理、知识库管理、模板管理、Skill 管理、知识审核、审计日志与使用报表的权限。
- **管理员后台位置**：**嵌入 Electron 客户端**，按登录角色显示。普通员工登录看不到管理员菜单；管理员登录后在设置界面看到额外的管理项。
- **业务逻辑**：
  - 用户通过账号密码登录，后端签发 JWT Token，前端后续请求携带 Token 鉴权。
  - 各厂商 LLM API Key 仅存储于后端环境变量，前端不可见。
  - 所有未授权请求返回 401，权限不足请求返回 403。

#### 5. 审计日志与使用报表 (Audit Log & Usage Report)

- **需求描述**：记录系统使用行为，支持管理员生成周报/月报。
- **审计日志内容**：用户 ID、操作时间、操作类型（普通问答 / RAG 查询 / 文件生成 / Skill 调用）、消耗 Token 数、是否成功。
- **保留期限**：审计日志保留 **12 个月**，超期数据由定时任务自动清理（或归档为 JSON 文件后删除），避免数据库无限增长导致报表查询变慢。
- **使用报表**：
  - 管理员可在后台按周 / 月维度生成报表，内容包括：活跃用户数、总对话次数、各功能使用占比、Token 消耗趋势、文件生成成功率、Skill 触发次数排行。
  - 报表支持导出为 Excel 文件。

#### 6. 业务工作流 Agent / Skill 系统

本节是 Project_R 区别于通用 AI 助手（如 Proma）的核心能力。

##### 6.1 三层 Skill 模型

| 层级 | 来源 | 可见范围 | 是否可改 |
|---|---|---|---|
| **官方 Skills** | 随软件预置（写在代码 `backend/skills/builtin/`） | 全员可用 | 不可修改 |
| **企业 Skills** | 管理员通过后台上传（`backend/skills/enterprise/`） | 全员可用 | 仅管理员可改 |
| **个人 Skills** | （暂不开放） | — | — |

##### 6.2 Skill 文件格式

沿用 Proma 的 Skill 格式（与 Claude Agent SDK 兼容）。每个 Skill 是一个目录，含 `SKILL.md` 描述触发条件、输入、步骤、输出。详见 `docs/agents/skills-design.md`。

##### 6.3 知识审核队列

- AI 在对话或会议纪要沉淀过程中提炼出的"规则知识"候选条目 → 写入 SQLite `knowledge_review` 表（不直接进 ChromaDB）。
- 管理员在后台"知识审核"页面查看候选条目 → 编辑润色 → 批准后入正式知识库。
- 不批准的候选条目可标记为"驳回"或"修改后重提"。

##### 6.4 Skill 调用流程

1. 当前阶段：用户在前端 Skill 面板手动选择 Skill，并继续输入具体要求
2. 后端通过 `selected_skill` 显式启动对应 Skill；长期目标再恢复自然语言触发语自动匹配
3. Skill 收集所需输入（项目信息 / 上传文件等），可分多轮交互
4. 执行 Skill 的处理步骤（调用 RAG、调用 LLM 总结、调用文件渲染）
5. 输出结果（聊天文本 / 文件下载 / 钉钉推送）
6. 写入审计日志

### 五、非功能性需求

#### 1. 性能与高并发指标

- 针对 20 人的高频并发调用，后端服务必须保持无状态（Stateless），会话数据统一由数据库层管理。
- 严禁在后台拉起具备人机交互等待（如 CLI 的 `Y/n` 确认）的进程，防止出现线程死锁与队列卡死。

#### 2. LLM Provider 多 Key 管理与降级

- 后端通过 `core/llm.py` 提供统一 LLM Provider 抽象，业务层只调用统一接口，不直接依赖某一家厂商 SDK。
- 第一版默认 Provider 为 Claude；同时预留 OpenAI / ChatGPT、DeepSeek、MiMo 等厂商配置。OpenAI、DeepSeek 与 MiMo 优先按 OpenAI-compatible Chat Completions 协议适配，Claude 按 Anthropic Messages API 适配。
- `model_profile` 是后端白名单模型配置档，用于把用户可见的模型版本映射到 provider、真实模型名和描述文案；它不等于 API Key。多个 profile 可共享同一 provider 的 Key 组，例如 DeepSeek Flash / Pro 共享 DeepSeek Key，MiMo V2.5 / V2.5-Pro 共享 MiMo Key。
- 聊天输入区允许展示 `/health/llm` 返回的已配置模型 profile；未配置 API Key 的模型不显示、不做占位。员工前端不得暴露 API Key、Key 数量以外的敏感细节、任意 Provider 配置或多 Key 管理界面。
- “思考模型”按钮不切换模型，而是向后端发送 `thinking` 布尔值。后端按 provider 生成实际参数：DeepSeek 使用 `thinking.type` + `reasoning_effort`，MiMo 使用 `thinking.type`；推理火力等配置保留在后端环境变量或 profile 中。
- 每个 Provider 独立配置多个 API Key，采用 **Round-Robin 轮询**策略：同一 Provider 内每次请求轮换下一个 Key，均匀分摊用量。
- 单个 Key 触发限流、网络异常或 5xx 错误时，自动尝试同 Provider 的下一个 Key；当前 Provider 所有 Key 均不可用时，系统向用户返回明确提示"AI 服务暂时不可用，请稍后重试"，不抛出原始错误信息。
- Provider 通过后端 `.env` 选择，默认读取 `LLM_PROVIDER`；必要时后端接口可接受管理员或调试场景传入的 `provider` 参数。员工前端只允许在聊天输入区选择后端白名单 `model_profile`，不得暴露渠道管理、API Key 或多 Key 配置。

#### 3. 跨平台兼容规范（Windows → Mac mini 迁移）

开发期在 Windows 单机上跑通整套系统，后续迁移到 Mac mini。所有代码必须遵守以下红线：

| 红线 | 错误做法 | 正确做法 |
|---|---|---|
| 路径分隔符 | 写死 `C:\Users\...` 或 `\\` | `pathlib.Path` 或 `os.path.join()` |
| 配置位置 | 写死 `D:/xxx` | 通过 `.env` 配置，默认相对路径 |
| 数据库位置 | 绝对路径 `D:/xxx.db` | 相对路径 + `.env` 覆盖 |
| 文件编码 | 使用系统默认 | 显式指定 `encoding='utf-8'` |
| 进程管理 | Windows 服务 / `taskkill` | 用 `uvicorn` 启动；迁移时 macOS 改用 launchd |
| 依赖包 | Windows-only | 仅选跨平台库 |
| 服务器 IP | 写死 `localhost` | 始终从配置读 IP，开发期 `127.0.0.1`，部署期 Mac mini IP |

依赖通过 `requirements.txt` + 纯净 `venv` 转移。

#### 4. 运行日志与监控方案

- **应用日志**：使用 Python `logging` 模块，按日滚动（`TimedRotatingFileHandler`），保留最近 30 天，存放于 `logs/` 目录。日志分级：INFO 记录正常请求，WARNING 记录降级事件，ERROR 记录异常与失败。
- **健康检查**：FastAPI 暴露 `/health` 与 `/health/llm` 端点，返回服务状态、向量库文件数、默认 Provider、各 Provider 是否已配置及 Key 数量；严禁返回 API Key 明文。
- **进程监控**：通过 LaunchDaemon 守护进程确保 FastAPI 崩溃后自动重启，无需人工介入（Windows 阶段使用 NSSM 或 Windows 服务作为临时方案）。
- **磁盘告警**：后台定时任务每小时检查磁盘使用率，超过 80% 时写入 WARNING 日志并可选发送内网告警通知。

#### 5. 运维与灾备机制（Mac mini 部署阶段）

- **固定 IP 绑定**：Mac mini 通过 `networksetup` 锁定静态局域网 IP（例：192.168.1.200）。
- **无人值守自启**：配置 `/Library/LaunchDaemons/` 守护进程与 `pmset` 机制，断电重启后无需人工登录即可自动拉起 FastAPI 后端进程。

#### 3.1 Phase 7 前端骨架要求

- Electron 主进程必须使用安全默认值：`contextIsolation: true`、`nodeIntegration: false`、最小化 `preload`，renderer 不直接访问 Node API。

## Phase 6 修订：基础对话接口

第六阶段的基础对话接口是后续 RAG、文件生成与 Skill 调度的通用消息底座。

- 后端提供会话创建、会话列表、会话详情、消息发送、消息历史分页读取、会话删除能力。
- user 消息与 assistant 消息均持久化到 `ChatMessage` 表，刷新客户端后可重新拉取历史。
- 每次发送消息时，后端先写入 user 消息，再读取当前会话最近 20 条成功消息作为 LLM 上下文。
- LLM 调用默认使用 `.env` 中的 `LLM_PROVIDER`；调试或管理员场景可在请求中传入 `provider`，员工前端默认不暴露模型和渠道选择。
- LLM 成功后写入 assistant 消息、Provider、模型与 Token 输入/输出/总量。
- LLM 失败时保留 user 消息，并写入脱敏失败记录；前端只收到统一错误提示，不暴露 API Key、请求头或供应商原始错误体。
- JSON 非流式接口先作为稳定基础能力；后续可增加独立 SSE 流式端点，不破坏当前接口。
- 基础对话审计需记录 Provider、模型、Token 输入/输出/总量、成功/失败状态；失败时记录脱敏后的错误摘要。
